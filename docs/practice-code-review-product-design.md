# 练习代码 Review 产品设计

## 背景

当前题目训练工作台已经支持用户从学习计划进入某道题的练习会话，在聊天页查看题面、和 AI 讨论思路，并手动把题目标记为已完成。这个闭环仍然缺少一个关键沉淀：用户真正写过什么代码、代码质量如何、哪里出错、改进后有没有变好。

代码 Review 记录应成为用户画像、错题复盘和能力评估的原始数据，而不是普通聊天记录的附属产物。聊天记录更适合表达过程，Review 记录更适合表达一次明确的代码提交、评分、问题归因和复盘结论。

## 产品目标

- 用户在题目练习聊天中粘贴完整 LeetCode 题解代码后，主模型应自主判断是否调用正式 Review 工具；用户确认后生成代码 Review 记录。
- 每道计划题支持多次 Review，记录用户多轮修改和分数变化。
- 每次 Review 保存代码快照、AI 点评、评分、是否及格和聊天上下文摘要。
- 题目完成资格由最近一次代码提交的 Review 结果决定。
- 前端主动引导用户粘贴完整代码，而不是只在用户点击完成时被动拦截。
- 为后续用户画像、错题复盘、薄弱标签分析和学习计划调整提供稳定数据来源。

## 非目标

- 不做在线运行、编译或判题。
- 不承诺 Review 分数等同于 LeetCode 真实通过结果。
- 不做完整用户画像页面。
- 不做 Review 版本 diff。
- 不做人工跳过完成门槛或人工 override 入口。
- 不把普通代码片段、报错片段、伪代码或语法咨询沉淀为正式 Review 记录。
- 不要求第一版支持所有编程语言的完美格式化，优先服务学习计划中使用的主要语言。

## 核心概念

### 代码提交

代码提交指用户在某个题目练习会话中粘贴的一段完整题解代码。它应该看起来可以作为当前题目的 LeetCode 提交，而不是只有局部片段、辅助函数、报错日志或伪代码。

主模型判断代码提交时需要结合当前题目信息。例如：

- `70. 爬楼梯` 对应常见入口函数 `climbStairs`。
- `5. 最长回文子串` 对应常见入口函数 `longestPalindrome`。

题目入口函数名用于提升识别准确率，但不能作为唯一判断依据。用户可能粘贴完整 `class Solution`、完整函数、带注释代码或不同语言代码。

### 有效 Review

有效 Review 是指一次已经被系统识别为当前题完整代码提交，并完成 AI Review 的记录。

只有有效 Review 会参与完成资格判断。普通聊天、代码片段讨论、不完整题解、报错分析和伪代码讨论都不参与完成资格判断。

### Review 版本

同一个用户在同一个 `计划 + 阶段 + 题目` 下可以生成多个 Review 版本。每次有效代码提交生成一个新版本，用于记录用户从初稿到修改后的演进过程。

版本之间不要求第一版展示 diff，但需要在产品上让用户看到每次提交的时间、分数、是否及格、代码快照和主要点评。

## 用户流程

### 自主触发 Review

```text
用户进入题目练习聊天页
  -> 聊天输入区提示用户可以粘贴完整 LeetCode 代码
  -> 用户发送包含代码的消息
  -> 主模型根据题目上下文自主判断是否像完整题解提交
  -> 如果不像完整提交，AI 正常回复并引导用户粘贴完整代码
  -> 如果像完整提交，AI 主动调用 submit_practice_code_review
  -> 系统弹出权限确认
  -> 用户允许后，Review 工具抽取代码、分析、打分并保存一条新的 Review 版本
  -> 用户拒绝或确认超时，不生成 Review 记录；AI 可继续普通点评，但不给正式分数
  -> Review 抽屉展示最新记录
  -> 完成按钮根据最新 Review 分数更新状态
```

### 标记完成

```text
用户点击题目已完成
  -> 系统检查最近一次有效 Review
  -> 没有有效 Review：禁止完成，并提示先粘贴完整代码完成 Review
  -> 最近一次有效 Review 未及格：禁止完成，并提示当前分数和改进要求
  -> 最近一次有效 Review 已及格：允许标记完成
```

完成资格只看最近一次代码提交对应的 Review。  
如果用户第一次 Review 得到 7 分，第二次提交后 Review 得到 5 分，则当前不能标记完成。这个规则保证完成状态反映用户最近一次代码质量，而不是历史最好成绩。

## 评分规则

### 总分

代码 Review 使用 10 分制。

```text
满分：10 分
及格线：6 分
完成条件：最近一次有效 Review 分数 >= 6
```

### 评分维度

第一版建议使用以下维度，便于后续画像聚合：

```text
正确性：0-4 分
复杂度：0-2 分
边界条件：0-2 分
代码质量：0-1 分
思路表达与题意贴合：0-1 分
```

总分为各维度之和。

如果代码存在明显正确性阻断问题，例如核心逻辑错误、无法覆盖基本样例、函数返回类型明显不符或缺少关键流程，则总分最高不应超过 5 分。

### 评分结论

每次 Review 至少给出以下结论：

- 总分。
- 是否及格。
- 主要扣分原因。
- 正确性评价。
- 复杂度评价。
- 边界条件评价。
- 代码质量评价。
- 下一次修改建议。

第一版完成资格只依赖分数是否达到 6 分，不额外引入复杂状态。后续如果需要，可以再扩展 `优秀`、`需复习`、`疑似非本题` 等标签。

## Review 记录内容

每条 Review 记录在产品语义上应包含：

- 所属用户。
- 所属学习计划。
- 所属阶段。
- 所属题目。
- 所属练习会话。
- Review 版本号。
- 用户原始消息引用。
- AI 回复消息引用。
- 用户粘贴的原始代码。
- 系统整理后的 LeetCode 提交格式代码。
- 编程语言。
- 触发识别的证据，例如题目入口函数、代码块语言或语言关键字。
- 本次 Review 前的聊天上下文摘要。
- 总分和各维度分。
- 是否及格。
- 扣分原因。
- 改进建议。
- 创建时间。

其中聊天上下文摘要用于把 Review 放回当时的学习过程里，例如用户之前卡在哪里、曾经问过什么边界条件、AI 已经提示过哪些方向。后续用户画像优先聚合 Review 记录，而不是重新分析完整聊天记录。

## 页面体验

### 聊天输入区引导

练习聊天页需要主动引导用户粘贴完整代码。文案应表达机制本身，而不是等用户点击完成后才告知。

建议文案：

```text
粘贴完整 LeetCode 代码后，AI 会自动生成 Review 记录并更新完成资格。
```

输入框 placeholder 可以根据状态动态变化：

```text
输入你的思路、问题，或粘贴完整 LeetCode 代码生成 Review...
```

### 完成按钮状态

完成按钮需要展示清晰的限制原因。

没有有效 Review 时：

```text
完成前需要先粘贴完整代码完成一次 AI Review。
```

最近一次 Review 未及格时：

```text
最近一次 Review 为 5/10，达到 6 分后可标记完成。
```

最近一次 Review 已及格时：

```text
标记为已完成
```

按钮禁用时仍应让用户能看到原因。桌面端可以用按钮旁说明或 hover tooltip，移动端应直接展示短说明。

### Review 抽屉

Review 抽屉从占位入口升级为版本历史。第一版应展示：

- 最新 Review 的分数和是否及格。
- 所有 Review 版本列表。
- 每个版本的提交时间、语言、总分、是否及格。
- 当前版本的代码快照。
- 当前版本的主要点评和改进建议。
- 最近一次 Review 对完成资格的影响。

抽屉空态文案：

```text
还没有 Review 记录。粘贴完整 LeetCode 代码后，AI 会自动生成 Review。
```

## 代码识别产品规则

代码提交判断由主模型基于系统 prompt 和工具描述自主完成。以下信号只作为 prompt 中可参考的判断依据，不作为后端规则识别路径。

### 规则预判

先用低成本规则判断用户消息是否疑似包含代码。判断信号包括：

- Markdown 代码块。
- 常见语言关键字，例如 `class Solution`、`public`、`def`、`function`、`return`。
- 当前题目的 LeetCode 入口函数名，例如 `climbStairs`、`longestPalindrome`。
- 括号、缩进、语句结构等代码形态。

规则预判只决定是否值得进一步识别，不直接决定是否生成正式 Review。

### 模型识别

当规则预判认为消息疑似包含代码时，系统再判断：

- 是否为代码提交。
- 是否属于当前题。
- 是否像完整 LeetCode 解法。
- 使用的编程语言。
- 原始代码范围。
- 可整理出的 LeetCode 提交格式代码。
- 不完整时缺少哪些部分。

只有当前题完整代码提交才生成有效 Review。  
如果用户只是粘贴片段、报错、伪代码或辅助函数，AI 应正常回答问题，并提示用户粘贴完整 `Solution` 代码以生成 Review 记录。

## 用户画像关系

Review 记录是用户画像的原始事实来源。后续画像可以从 Review 记录聚合，而不是直接分析聊天历史。

可聚合维度包括：

- 标签维度平均分。
- 难度维度平均分。
- 首次 Review 及格率。
- 平均需要几次 Review 达到及格。
- 常见扣分原因。
- 正确性、复杂度、边界条件、代码质量的长期趋势。
- 某个阶段或某类题目的薄弱点。

第一版只负责把 Review 数据记录完整，不做画像页面。

## 成功标准

- 用户粘贴完整题解代码后，能自动得到 Review 记录。
- 同一道计划题能生成多个 Review 版本。
- Review 抽屉能清楚展示版本、代码快照、分数和点评。
- 没有有效 Review 时不能完成题目。
- 最近一次有效 Review 低于 6 分时不能完成题目。
- 最近一次有效 Review 达到 6 分时可以完成题目。
- 普通代码片段或非完整题解不会污染正式 Review 版本。
