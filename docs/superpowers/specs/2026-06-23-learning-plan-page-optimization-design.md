# 学习计划页面优化设计

## 背景

当前学习计划页面已经支持创建草案、确认正式计划、列表和详情展示，但页面仍偏向“左侧列表 + 右侧创建/详情”的早期形态。用户希望将计划页改成更清晰的管理页面：上方展示创建入口与汇总信息，下方分页展示计划列表；计划可以删除；新建计划使用问卷式表单，并为用户保留自然语言补充空间。

本设计聚焦学习计划管理页和创建流程优化，不扩展计划执行、打卡、进度追踪或正式计划编辑。

## 目标

- 将计划页调整为两张主要卡片：顶部汇总与创建入口，下方分页计划列表。
- 去掉搜索框和搜索按钮，页面主动作只保留“新建计划”。
- 列表行左侧展示计划内容，右侧展示状态和操作。
- 新建计划使用弹窗问卷，一次性展示所有问题。
- 草案生成后在页面内预览，不在弹窗内承载长内容。
- 支持正式计划删除，并保证只能删除当前用户自己的计划。
- 将列表接口升级为后端分页响应，避免前端假分页和统计不准。
- 优化问卷字段，让用户少填开放目标，由模型生成目标摘要，用户后续可基于摘要再生成。

## 非目标

- 不做正式计划的直接编辑保存。
- 不做计划执行进度、完成状态、每日任务或打卡。
- 不做搜索、筛选和排序。
- 不引入单独的目标摘要生成接口。
- 不让弹窗承载完整草案预览。
- 不扩展题库标签管理后台。

## 页面结构

页面采用上下两块主要区域。

### 顶部汇总卡片

左侧展示动态汇总：

- `当前共有 N 个计划`
- `进行中 N`
- `已归档 N`
- `最近创建 yyyy-MM-dd`，没有计划时显示空状态文案

右侧展示唯一主按钮：

- `新建计划`

桌面端按钮右对齐；移动端卡片换行展示，汇总在上，按钮在下。按钮可右对齐或全宽，具体以现有响应式样式稳定性为准。

### 计划列表卡片

列表卡片展示分页数据。每行采用左右布局：

- 左侧：标题、目标摘要、周期、每周投入、语言、主题等元信息。
- 右侧：状态 badge、查看按钮、删除按钮。

当前选中的计划行高亮。查看计划后，详情区域显示在列表下方或同页主内容区；不使用弹窗展示详情。

空状态文案聚焦下一步动作，例如“暂无正式计划，先新建一个学习计划。”

## 新建计划流程

1. 用户点击顶部卡片右侧的 `新建计划`。
2. 页面打开弹窗，展示问卷式表单。
3. 用户提交后，弹窗进入生成中状态，关闭和提交按钮禁用。
4. 后端创建草案并生成计划。
5. 草案生成成功后关闭弹窗，页面内展示草案预览。
6. 用户确认草案后保存为正式计划。
7. 前端刷新第一页列表，并选中新创建的计划。

如果生成失败，弹窗保持打开并展示错误，用户可修改问卷后重试。

如果用户关闭已填写内容的弹窗，前端提示确认放弃。

## 问卷字段设计

问卷不再提供独立的“学习目标”必填输入。目标由结构化选项、补充想法和模型生成结果共同形成。

### 计划场景

必填，使用单选卡片或分段控件：

- 面试冲刺：映射 `INTERVIEW_SPRINT`
- 专项突破：映射 `TOPIC_BREAKTHROUGH`
- 基础巩固：映射 `PRACTICE_GOAL`
- 错题复盘：映射 `MISTAKE_REVIEW`
- 长期学习：映射 `LONG_TERM_LEARNING`

不再单独展示“是否面试导向”。是否面试导向由计划场景推导：选择“面试冲刺”时提交 `interviewOriented = true`，其他场景提交 `false`。

### 时间投入

必填：

- 周期，默认 4 周，必须为正整数。
- 每周投入，默认 6 小时，必须为正整数。

### 当前水平

必填：

- 初级：`BEGINNER`
- 中级：`INTERMEDIATE`
- 高级：`ADVANCED`

默认中级。

### 编程语言

必填，下拉框，默认 Java。

第一版内置算法刷题常用语言：

- Java
- Python3
- C++
- JavaScript
- TypeScript
- Go
- C#
- C
- Kotlin
- Swift
- Rust

该列表参考 LeetCode 官方语言环境说明和常见题解语言，第一版不加入 MySQL、Bash 等专项语言，避免算法计划页出现不相关选项。LeetCode 官方环境说明见：`https://support.leetcode.com/hc/en-us/articles/360011833974-What-are-the-environments-for-the-programming-languages`。

### 难度分布

不使用简单/中等/困难单选。改为一个滑块或分段滑块，映射到固定难度比例。

第一版建议提供三个稳定档位：

- 入门：简单 60% / 中等 35% / 困难 5%
- 均衡：简单 25% / 中等 55% / 困难 20%
- 冲刺：简单 10% / 中等 55% / 困难 35%

UI 可以用一个滑块在三档之间切换，并实时显示比例。这样避免用户误解“选择简单就是全部简单”，也避免三段独立百分比输入带来的总和校验问题。

后端当前只有 `difficultyPreference` 枚举。第一版实现时可将三档映射到现有枚举以减少后端改动：

- 入门 -> `EASY`
- 均衡 -> `MIXED`
- 冲刺 -> `HARD`

同时在自动生成的 `goal` 或补充上下文中写入具体比例，供模型生成计划时参考。后续若要更强约束，再新增结构化 `difficultyDistribution` 字段。

### 主题偏好

可选，多选标签。默认可以不选，由系统根据计划场景和题库情况安排。

第一版内置常见算法标签：

- 数组
- 哈希表
- 字符串
- 双指针
- 滑动窗口
- 栈
- 队列
- 链表
- 二叉树
- 图
- DFS/BFS
- 二分查找
- 动态规划
- 贪心
- 堆
- 回溯
- 位运算

如果计划场景为“专项突破”，主题偏好应至少选择一个；否则专项场景缺少核心约束。其他场景不要求选择主题。

提交到后端时，标签值需要映射到题库中已有英文标签或当前学习计划生成逻辑可识别的标签，例如 `Array`、`Hash Table`、`Dynamic Programming`。

### 补充想法

可选，多行输入，放在问卷最后。

用于表达结构化字段覆盖不到的信息，例如：

- 希望少安排数学题。
- 下个月有后端面试。
- 想每周留一天复盘。
- 动态规划需要多一点讲解。

该字段不作为独立后端字段保存。第一版提交时由前端拼入自动生成的 `goal` 文本。

## 目标摘要设计

用户不在问卷阶段手写学习目标。系统在生成草案时，基于问卷生成用户可读的目标摘要。

示例：

```text
用 4 周时间，以 Java 为主要语言，面向后端算法面试进行中等强度训练。计划重点覆盖数组、哈希表和动态规划，并通过阶段复盘提升解题稳定性。
```

目标摘要展示在草案预览顶部，对应现有 `draftPlan.goal` 的用户可见语义。

用户可以在草案预览阶段修改目标摘要。修改后不直接更新正式计划，而是提示需要重新生成草案。用户点击“按新目标重新生成”后，前端通过现有 draft message 流程发送修改后的目标摘要，让后端重新生成草案。

语义边界：

- `goal`：用户可见、可修改的目标摘要。
- `profileSummary`：系统生成的学习画像摘要，只读展示。

第一版不新增单独的“总结目标”接口。目标摘要随草案一起生成，减少一次额外模型等待。

## API 设计

### 分页列表

将当前 `GET /api/learning-plans` 从数组响应升级为分页响应。

请求：

```http
GET /api/learning-plans?page=1&pageSize=10
```

约束：

- `page` 从 1 开始，默认 1。
- `pageSize` 默认 10，最大 50。
- 列表按 `created_at DESC, id DESC` 排序。
- 只返回当前登录用户的计划。

响应 `data`：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "pageSize": 10,
  "activeCount": 0,
  "archivedCount": 0,
  "latestCreatedAt": null
}
```

前端顶部汇总使用 `total`、`activeCount`、`archivedCount`、`latestCreatedAt`，而不是只根据当前页 `items` 推导，避免统计不准。

### 删除计划

新增：

```http
DELETE /api/learning-plans/{planId}
```

约束：

- 只允许删除当前登录用户自己的计划。
- 删除不存在或不属于当前用户的计划时，返回学习计划不存在错误。
- 删除成功返回 `ApiResponse<Void>`，`data` 为空。

删除持久化注意事项：

- `learning_plan_phase.plan_id` 已设置 `ON DELETE CASCADE`。
- `learning_plan_recommended_problem.phase_id` 已设置 `ON DELETE CASCADE`。
- `learning_plan_draft.confirmed_plan_id` 引用 `learning_plan(id)`，当前迁移未设置级联或置空策略。删除正式计划前需要先将当前用户相关 draft 的 `confirmed_plan_id` 置空，或新增迁移将该外键改为 `ON DELETE SET NULL`。

第一版推荐在服务层删除前显式置空相关 draft 引用，再删除正式计划，避免调整已发布外键策略带来的迁移风险。

### 草案创建

保留：

```http
POST /api/learning-plans/drafts
```

由于现有请求仍需要 `goal`，前端根据问卷自动组装 `goal`：

```text
计划场景：面试冲刺
周期：4 周
每周投入：6 小时
当前水平：中级
编程语言：Java
难度分布：简单 25%，中等 55%，困难 20%
主题偏好：Array, Hash Table, Dynamic Programming
补充想法：希望每周留一天复盘。
```

`difficultyPreference` 第一版按难度分布档位映射到现有枚举。

## 前端状态设计

建议将现有计划页状态拆成更清晰的页面状态：

- `plansPage`：当前分页响应。
- `page` / `pageSize`：分页参数。
- `selectedPlan`：当前查看的正式计划详情。
- `draft`：当前草案。
- `isCreateModalOpen`：创建弹窗是否打开。
- `createSubmitting`：问卷提交和草案生成中。
- `deletingPlanId`：当前删除中的计划 ID。
- `error`：页面级错误。
- `modalError`：创建弹窗错误。

删除流程：

1. 用户点击行右侧删除。
2. 前端展示二次确认。
3. 确认后禁用该行删除按钮。
4. 调用删除接口。
5. 成功后刷新当前页。
6. 如果当前页删除后为空且不是第一页，页码回退一页并刷新。
7. 如果删除的是当前选中计划，清空详情或选中刷新后第一条。

确认草案流程：

1. 用户确认草案。
2. 后端返回 `planId`。
3. 前端关闭草案预览状态。
4. 刷新第一页列表。
5. 加载并选中新 `planId` 的详情。

## 组件设计

建议在 `frontend/src/learning-plans` 下拆分：

- `LearningPlanSummaryCard.tsx`：顶部汇总与新建按钮。
- `LearningPlanListCard.tsx`：分页列表、行操作和分页控件。
- `LearningPlanCreateModal.tsx`：问卷弹窗。
- `DifficultyDistributionControl.tsx`：难度滑块与比例展示。
- 复用 `LearningPlanDraftPanel.tsx` 和 `PlanPreview.tsx` 展示草案和详情。

现有 `LearningPlanWizard.tsx` 可以被替换为 `LearningPlanCreateModal.tsx`，或者先保留文件名但改为一次性问卷。由于交互语义已不再是分步 wizard，推荐新建组件并逐步移除旧 wizard。

## 后端实现边界

`mentor-application`：

- `LearningPlanRepository` 增加分页查询、统计汇总、删除和置空 draft 引用能力。
- `LearningPlanService` 增加分页列表和删除用例。
- 删除用例负责用户归属校验和异常映射。

`mentor-api`：

- Controller 增加分页参数和删除端点。
- DTO 增加 `LearningPlanPageResponse`。
- API contract 常量类增加删除和分页相关公共契约字段时应复用现有常量风格。

MyBatis：

- 增加分页查询 SQL。
- 增加总数、状态统计、最近创建时间查询。
- 增加置空 confirmed draft 引用 SQL。
- 增加按 `id` + `user_id` 删除正式计划 SQL。

## 错误与边界状态

- 列表加载失败：在列表卡片内展示错误，可保留顶部汇总为空态。
- 删除失败：保留当前列表，展示页面级错误。
- 草案生成失败：弹窗内展示错误，不关闭弹窗。
- 草案确认失败：草案预览保留，展示页面级错误。
- 分页越界：后端返回空页；前端在删除导致空页时主动回退。
- 专项突破未选择主题：前端阻止提交并提示选择至少一个主题。
- 周期和每周投入非正整数：前端阻止提交。

## 测试计划

前端：

- 顶部汇总卡片正确展示 total、activeCount、archivedCount 和 latestCreatedAt。
- 搜索框和搜索按钮不再出现。
- 计划列表分页渲染，行内容左侧、状态和操作右侧。
- 删除按钮触发二次确认、成功刷新、失败提示。
- 删除当前页最后一条时回退上一页。
- 删除接口通过 `apiFetch` 发送，非 GET 请求正确附加 CSRF token。
- 新建按钮打开弹窗。
- 问卷提交 payload 包含自动组装的 goal、语言、主题、难度偏好。
- 专项突破未选择主题时阻止提交。
- 草案生成成功后关闭弹窗并展示页面内预览。
- 确认草案后刷新第一页并选中新计划。
- 目标摘要修改后通过追问流程触发重新生成。

后端：

- 分页列表按当前用户过滤并返回汇总统计。
- page/pageSize 默认值和最大值生效。
- 删除不存在或其他用户计划返回 not found。
- 删除计划前处理 confirmed draft 引用。
- 删除计划后关联 phase 和 recommended problem 被清理。

## 风险与取舍

- 将列表响应从数组改为分页对象会影响现有前端测试和调用点，需要同步改类型、mock 和断言。
- 难度分布第一版通过 `difficultyPreference` + goal 文本表达，不是强结构化约束；模型可能不会严格按比例选题。后续可增加 `difficultyDistribution` 字段强化约束。
- 目标摘要可编辑后重新生成会复用追问接口，用户体验上应明确“修改目标需要重新生成草案”，避免误以为只是改标题。
- 删除正式计划涉及 draft 外键引用，需要先处理引用再删除，否则数据库会拒绝删除。
