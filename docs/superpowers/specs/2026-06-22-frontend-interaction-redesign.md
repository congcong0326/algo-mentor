# 前端交互重设计规格

## 背景

`algo-mentor` 当前前端已经具备 AI SSE 调试、题库和学习计划三个页面能力，但整体仍偏“测试台外壳 + Tab 切换”。学习计划页把正式计划列表、新建表单、Agent 追问、草案预览和正式计划详情放在同一屏，信息密度高但主流程不清晰。

本次设计目标是先重做前端交互，不新增后端接口或数据库结构。现有后端接口已经支持学习计划草案创建、Agent 追问、草案确认、计划列表和计划详情，足够支撑一次前端体验重组。

## 决策摘要

- 登录页独立：未登录用户进入 `/login`，登录成功后进入应用。
- 登录后使用顶部导航布局，不使用侧边栏。
- 顶部导航保留三个入口：学习计划、题库、AI 调试。
- 登录后的默认页面为学习计划。
- 学习计划页优化重点放在新建计划体验。
- 新建计划采用分步向导，不采用对话优先或双栏工作台方案。
- 本次不新增计划进度、任务完成状态、归档、删除、当前周视图等后端能力。

## 范围

### 本次包含

- 新增独立登录页。
- 新增登录后应用外壳，提供顶部导航、用户信息和退出操作。
- 将学习计划设为登录后的默认页面。
- 将 AI SSE 调试保留为导航中的工具页。
- 将学习计划的新建流程改为分步向导。
- 优化 Agent 追问、草案预览和确认保存的状态衔接。
- 保持题库页面现有搜索、筛选、列表和详情体验。

### 本次不包含

- 不新增后端 API。
- 不修改学习计划数据库模型。
- 不实现计划任务打卡、进度统计、归档、删除或编辑正式计划。
- 不重做题库详情阅读器或 AI SSE 调试功能。

## 整体信息架构

### 未登录状态

未登录用户访问应用时进入 `/login`。登录页只承担认证入口职责，内容包括：

- 产品名 `Algo Mentor`。
- 一句简短说明，表达这是算法学习和计划生成工具。
- Google 登录按钮，跳转现有 OAuth2 登录入口。
- 登录失败提示，例如 URL 中包含认证失败参数时展示错误文案。

未登录访问受保护页面时，由应用外壳检查 `/api/auth/me` 的结果。如果返回 401，则导向登录页。

### 登录后状态

登录后进入统一应用外壳。外壳包含：

- 顶部左侧品牌名。
- 品牌名右侧的顶部导航：学习计划、题库、AI 调试。
- 顶部右侧当前用户信息和退出按钮。
- 主内容区渲染当前页面。

默认激活页面为学习计划。AI 调试不再承担首页角色，但继续保留完整 SSE 测试能力。

## 学习计划新建流程

学习计划页默认展示计划列表和明确的新建入口。点击“新建计划”后进入分步向导。

### 第一步：目标

用户选择计划意图并填写学习目标。

计划意图继续使用现有枚举：

- 面试冲刺
- 刷题目标
- 专题突破
- 长期学习
- 能力诊断
- 错题复盘

目标输入是必填项。页面可以提供简短示例，但不引入复杂模板系统。

### 第二步：时间与水平

用户填写结构化参数：

- 周期，最小值为 1。
- 每周投入小时数，最小值为 1。
- 当前水平：入门、中级、高级。
- 编程语言，默认可继续使用 `Java`。
- 难度偏好：Easy、Medium、Hard、Mixed。
- 是否面试导向。

这些字段映射到现有 `LearningPlanCreateDraftRequest`。

### 第三步：主题偏好

主题偏好从普通字符串输入改为标签输入体验。用户可以通过逗号、中文逗号、空格或回车拆分标签。

示例标签包括 `Array`、`Hash Table`、`Dynamic Programming`。标签列表最终仍转换为现有 `topicPreferences: string[]`。

### 第四步：生成与确认

点击“生成草案”时调用 `createLearningPlanDraft`。

如果返回 `COLLECTING`：

- 展示 Agent 追问。
- 展示回答输入框。
- 点击发送后调用 `sendLearningPlanDraftMessage`。
- 返回结果继续按状态更新界面。

如果返回 `GENERATED`：

- 展示草案预览。
- 用户点击“确认保存”后调用 `confirmLearningPlanDraft`。
- 确认成功后刷新学习计划列表。
- 自动选中新生成的正式计划并展示详情。

如果返回失败、过期或接口错误：

- 在当前流程顶部展示错误。
- 保留用户已填写的向导内容。
- 允许用户回到前面步骤调整后重新生成。

## 页面状态模型

学习计划页使用明确的流程状态：

- `idle`：浏览正式计划。
- `creating`：填写新建计划向导。
- `generating`：提交创建草案请求中。
- `collecting`：Agent 正在追问补充信息。
- `previewing`：草案已生成，等待确认保存。
- `confirming`：确认保存中。

列表、详情和草案状态分开维护。正式计划详情使用 `selectedPlan`，草案流程使用 `draft` 和本地向导表单状态。确认成功后清空 `draft` 和创建流程状态。

## 数据流

进入学习计划页：

1. 调用 `getLearningPlans`。
2. 如果列表不为空，可默认选中第一条正式计划。
3. 如果列表为空，展示空状态和新建入口。

新建学习计划：

1. 用户填写向导，本地维护表单草稿。
2. 第四步调用 `createLearningPlanDraft`。
3. 使用返回的 `draftId` 继续追问或确认。
4. `COLLECTING` 时调用 `sendLearningPlanDraftMessage`。
5. `GENERATED` 时展示 `draftPlan`。
6. 确认时调用 `confirmLearningPlanDraft`。
7. 确认成功后调用 `getLearningPlans` 和 `getLearningPlanDetail` 刷新正式计划视图。

题库和 AI 调试继续使用现有数据流，只纳入应用外壳。

## 异常与边界处理

- `/api/auth/me` 返回 401 时进入登录页。
- 非认证错误在页面顶部展示简短错误信息。
- 生成、追问和确认过程中禁用相关按钮，避免重复提交。
- 必填目标为空时禁用生成入口。
- 数字字段低于最小值时阻止进入下一步。
- 计划列表为空时展示空状态和新建按钮。
- 草案生成失败或过期时允许用户回到向导修改后重新提交。
- 退出登录失败时在应用外壳内展示错误，不清空当前页面数据。

## 组件拆分

- `AppShell`：登录后顶部导航、当前用户、退出、页面切换。
- `LoginPage`：独立登录页，提供 Google 登录入口和失败提示。
- `LearningPlans`：学习计划页面容器，加载计划列表、维护选中计划和创建流程状态。
- `LearningPlanWizard`：分步向导表单，输出 `LearningPlanCreateDraftRequest`。
- `LearningPlanDraftPanel`：处理 Agent 追问、草案预览和确认保存。
- `LearningPlanDetail`：展示正式计划详情。
- `PlanPreview`：展示草案和正式计划共用的阶段、目标、标签和题目结构。
- `ProblemLibrary`：保留现有题库交互，接入应用外壳。
- `AiDebugConsole`：从当前 `App` 中拆出 AI SSE 调试页，接入应用外壳。

## 响应式行为

桌面端使用顶部导航和宽内容区。学习计划创建流程以单列步骤内容为主，避免在同一屏塞入过多表单。

移动端顶部栏保留品牌名和用户操作，主导航移动到顶部栏下方的横向三项导航。向导步骤保持单列，底部提供上一步、下一步、生成草案等主操作。

## 测试范围

优先覆盖以下前端交互：

- 未登录时显示登录页。
- 登录后默认进入学习计划。
- 顶部导航能切换学习计划、题库、AI 调试。
- 学习计划向导能填写目标、时间、水平和主题偏好。
- 向导提交时生成符合 `LearningPlanCreateDraftRequest` 的请求。
- `COLLECTING` 状态展示 Agent 追问和回答输入。
- `GENERATED` 状态展示草案预览和确认按钮。
- 确认成功后刷新列表并选中新计划。
- 接口错误会在当前流程展示错误信息。

题库和 AI 调试以不回归为主，验证纳入应用外壳后核心交互仍可用。

## 实施约束

- 前端继续使用 React + TypeScript + Vite。
- 不新增 UI 库，优先沿用现有 CSS 和 lucide-react 图标。
- 公共页面标识、视图 key、路由路径等跨组件契约应集中到常量或类型中。
- 本次设计文档确认后，再进入实施计划编写阶段。
