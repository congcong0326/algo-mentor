# Agent 跨轮次上下文中的工具结果设计

更新时间：2026-06-29

## 核心结论

ReAct / tool-calling Agent 中，工具结果要分三层看：

- **协议链路**：当前工具调用回合内，`assistant tool_call -> tool result -> assistant final` 必须按协议保留。
- **持久化记录**：原始工具结果应该保存到 trace、tool result store 或业务表，服务于审计、回放、调试和业务沉淀。
- **下一轮模型上下文**：应保留工具调用链路和必要工具结果，但不应默认把所有历史工具结果原样带入。应根据价值、大小、敏感性和时效性选择原文、摘要、占位符或引用 ID。

一句话：**工具结果应保存，但不等于每一轮都原样喂给模型。**

## 正确的工具消息顺序

如果跨轮次选择带上原始工具结果，必须保持真实的 tool-calling transcript 顺序，不能把工具结果集中挪到系统提示词后面。

典型顺序是：

```text
developer/system
user: 用户问题 1
assistant: tool_call(call_1)
tool: tool_result(call_1)
assistant: 基于工具结果给最终回答
user: 用户问题 2
assistant: 新回答或新 tool_call
```

如果一次 assistant message 并行发起多个工具调用，顺序通常是：

```text
developer/system
user: 用户问题
assistant: tool_call(call_1), tool_call(call_2)
tool: tool_result(call_1)
tool: tool_result(call_2)
assistant: final answer
```

关键点：

- `tool result` 不能没有对应的 `assistant tool_call`。
- `tool result` 不能随意移动到稳定系统规则之后作为普通上下文。
- 如果要重排、压缩或改写，改写后的内容就不再是 tool role，而应变成摘要、memory、runtime context 或 resultRef。

## 不同 API 中的工具结果形态

不同模型 API 对工具调用结果的 wire format 不一样，但语义是一致的：模型先发起工具调用，应用执行工具，再把工具结果按同一个调用 ID 回传给模型。

### Claude Messages API

Claude Messages API 里，模型返回的工具调用是 assistant 消息中的 `tool_use` block：

```json
{
  "role": "assistant",
  "content": [
    {
      "type": "tool_use",
      "id": "toolu_xxx",
      "name": "search_problem",
      "input": {
        "keyword": "binary search"
      }
    }
  ]
}
```

工具结果不是单独的 `tool` role，而是放在下一条 `role: "user"` 消息的 `content` block 中：

```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_xxx",
      "content": "..."
    }
  ]
}
```

所以 Claude 的形式可以理解为：

```text
assistant content: tool_use(id)
user content: tool_result(tool_use_id)
```

这里虽然工具结果在 `user` 消息中，但它不是普通用户自然语言，而是协议化的工具结果 block。

### OpenAI Chat Completions API

OpenAI Chat Completions 中，工具调用是 assistant 消息里的 `tool_calls` 字段：

```json
{
  "role": "assistant",
  "tool_calls": [
    {
      "id": "call_xxx",
      "type": "function",
      "function": {
        "name": "search_problem",
        "arguments": "{\"keyword\":\"binary search\"}"
      }
    }
  ]
}
```

工具结果使用单独的 `role: "tool"` 消息，并通过 `tool_call_id` 关联：

```json
{
  "role": "tool",
  "tool_call_id": "call_xxx",
  "content": "..."
}
```

所以 Chat Completions 的形式可以理解为：

```text
assistant message: tool_calls[].id
tool message: role=tool, tool_call_id
```

和 Claude 相比，它主要是多了一个显式的 `tool` role；语义上仍然是 tool call 和 tool result 成对出现。

### OpenAI Responses API

OpenAI Responses API 更偏向 item / event 结构。模型输出工具调用时是 `function_call` item：

```json
{
  "type": "function_call",
  "call_id": "call_xxx",
  "name": "search_problem",
  "arguments": "{\"keyword\":\"binary search\"}"
}
```

工具执行后，下一次请求中回传 `function_call_output` item：

```json
{
  "type": "function_call_output",
  "call_id": "call_xxx",
  "output": "..."
}
```

所以 Responses API 的形式可以理解为：

```text
output item: function_call.call_id
input item: function_call_output.call_id
```

这里没有 `role: "tool"`，而是用不同 item type 和同一个 `call_id` 维护配对关系。

### 生命周期结论

三种协议形态不同，但生命周期约束一致：

- `tool_use` / `tool_calls` / `function_call` 和对应结果必须配对。
- 如果跨轮保留原始 transcript，应按原协议形态保留工具结果。
- 如果删除历史工具调用，通常应同时删除对应工具结果，避免孤立消息。
- 如果清理历史工具结果，优先保留 block / message / item 结构，只替换内容为摘要、占位符或 resultRef。
- 如果工具结果已经被抽象成普通摘要、memory、业务状态或运行时上下文，就不应该再伪装成协议级 tool result。

因此，跨轮是否“带工具结果”的更准确表述是：

```text
应该保留工具调用链路和必要工具结果；
原始结果要按协议 role / block / item 形态保留；
被摘要化、裁剪、引用化或业务事实化后，应作为普通上下文处理。
```

## 为什么不能无脑跨轮次带完整工具结果

当前 ReAct 回合内，工具结果必须回传给模型，因为模型需要它完成推理和最终回答。

但当这一轮已经结束，用户发起下一条消息时，历史工具结果的价值会下降。无脑带入完整工具结果有几个问题：

- 工具结果可能只是过程数据，例如搜索候选、读取文件片段、查过滤条件、拉日志。
- assistant final 已经消化过工具结果，下一轮通常只需要结论。
- 工具结果可能很大，会占用上下文窗口并增加成本。
- 工具结果可能过期，例如外部搜索、状态查询、价格、权限、运行结果。
- 工具结果可能包含敏感信息，不适合长期作为普通历史反复传给模型。
- 动态工具结果放入上下文越靠前，对 prompt caching 越不友好。

因此，更合理的做法是：

```text
原始工具结果：保存到 trace / store / 业务表
活跃上下文：只带对当前回答必要的信息
大结果：带摘要 + resultRef，需要时再读
```

## Codex 的公开设计

Codex 官方手册说明，一个 thread 是一次会话，包含用户 prompt、模型输出和随后的 tool calls。一个 thread 可以包含多次用户 prompt，也可以恢复继续。

Codex 在工作过程中会收集：

- 文件内容。
- 工具输出。
- 已经做了什么、还需要做什么的持续记录。

同时，Codex 也受模型 context window 限制。官方手册说明，长任务中 Codex 可能自动 compact：总结相关信息并丢弃不太相关的细节。配置示例中也有：

```text
model_auto_compact_token_limit
tool_output_token_limit
compact_prompt
experimental_compact_prompt_file
```

这说明 Codex 的公开设计并不是“每轮都无限原样重放所有工具输出”，而是：

```text
完整线程/事件：可持续记录、可恢复
活跃模型上下文：受窗口和工具输出预算管理，必要时压缩
```

需要注意：Codex 并未公开底层每次 API 请求如何逐条组装历史 tool messages，因此不能断言其内部总是原样携带或总是清理。能确定的是，公开文档支持“线程包含工具调用和工具输出，但长上下文会自动压缩”的设计方向。

参考：

- OpenAI Codex Prompting: https://developers.openai.com/codex/prompting
- OpenAI Codex Configuration Reference: https://developers.openai.com/codex/config-reference

## Claude / Anthropic 的公开设计

Claude API 的工具调用协议更明确：模型返回 `tool_use` 后，应用执行工具，再把 `tool_result` 作为后续消息发回模型。未完成的工具调用链必须保持配对。

但在长会话层面，Anthropic 提供了上下文管理能力：

- **Compaction**：当上下文接近限制时，对旧上下文做总结。
- **Context editing**：在请求进入 Claude 前编辑上下文。
- **Tool result clearing**：清理旧工具结果，并用占位内容替代，让模型知道结果已被移除。

这说明 Claude 的推荐方向也是：

```text
当前 tool_use/tool_result 回合：按协议保留
长历史：允许清理、压缩、占位
客户端完整历史：可以继续保留
模型实际上下文：可以是被编辑过的版本
```

参考：

- Anthropic Tool use: https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/implement-tool-use
- Anthropic Context editing: https://docs.anthropic.com/en/docs/build-with-claude/context-editing
- Anthropic Compaction: https://docs.anthropic.com/en/docs/build-with-claude/compaction
- Claude Code settings: https://docs.anthropic.com/en/docs/claude-code/settings

## Claude Code 2.1.195 的实现观察

本节基于 `@anthropic-ai/claude-code@2.1.195` 和 `@anthropic-ai/claude-code-linux-x64@2.1.195` 的 npm 分发包阅读。公开 GitHub 仓库主要包含文档、脚本、插件示例和安装说明；核心运行逻辑在平台 native binary 中。该 binary 内包含压缩后的 JavaScript bundle，因此以下变量名来自压缩产物，但行为路径可以从字符串和函数调用关系确认。

结论：**Claude Code 并不是简单地在每一轮请求中原样重放所有历史 `tool_result`。它会保留 transcript 结构，但会在多个阶段把大型或旧的工具结果替换成持久化引用、预览或占位符。**

### 1. 工具执行当下会先做大结果持久化

工具返回结果进入 `tool_result` block 前，会经过结果大小处理逻辑：

- 空结果会被改成类似“工具完成但没有输出”的短文本。
- 含图片或文档的结果不会走普通文本持久化路径。
- 文本或 text block 数组过大时，会写入会话级 `tool-results` 目录，文件名按 `tool_use_id` 保存为 `.txt` 或 `.json`。
- 进入模型上下文的 `tool_result.content` 变成 `<persisted-output>...</persisted-output>` 包裹的说明，包含完整文件路径和约 2000 字符预览，而不是完整原文。

压缩产物中对应的关键路径大致是：

```text
Wdt / gIa -> hIa -> pDe -> fDe
```

其中可见默认阈值包括：

- `qca = 400000`：普通工具结果持久化的默认字符阈值。
- `Gdt = 2000`：持久化提示中预览内容长度。
- `Pdo = "tool-results"`：会话内工具结果目录名。
- `dDe = "<persisted-output>"` / `Mdo = "</persisted-output>"`：持久化结果提示的包裹标记。

这意味着即使是当前 run 内的工具结果，也不一定把完整 raw output 直接塞回模型；大输出会被变成“文件引用 + 预览”。

### 2. 每次主循环请求前会做 message-level 工具结果预算控制

Claude Code 主循环在每次调用模型前，会先复制当前消息列表，然后对消息做一次工具结果替换：

```text
ce = [...Py(messages)]
ce = await AIa(ce, contentReplacementState, ...)
```

`AIa` 内部调用 `vvp`，扫描历史消息里的 `tool_result`：

- 只处理 `user` 消息中的 `tool_result` block。
- 跳过已经是 `<persisted-output>` 或 `[Old tool result content cleared]` 的内容。
- 跳过包含图片或文档的结果。
- 按 assistant 回合分组，判断该组后续工具结果的大小。
- 如果单组超过约 `Vca = 200000` 字符，会优先选择最大的 fresh tool result 持久化。
- 替换时不删除 `tool_result` block，只把对应 `tool_result.content` 改成 `<persisted-output>` 提示。
- `contentReplacementState` 会记录已经见过的 `tool_use_id` 和替换内容，后续轮次会复用同一个替换，避免同一历史结果反复变回 raw output。

这条路径说明 Claude Code 在“下一轮模型上下文”层面主动做瘦身：历史消息仍按真实 transcript 保留，但大工具结果会被引用化。

### 3. context hint 兜底会清理旧工具结果，但保留最近结果

Claude Code 还有一条和服务端 `context_hint` 相关的兜底路径。它不是直接调用公开文档里的 `context_management.edits.clear_tool_uses_20250919`，而是在符合条件时给请求加 `context_hint`；如果服务端提示上下文需要处理或请求失败，再本地执行旧工具结果清理。

对应路径大致是：

```text
createContextHintController -> buildRequestParams
handleHintReject / onStreamFallback -> applyHintEdits
applyHintEdits -> dca -> hao -> Ujt
```

关键规则：

- 只在主 REPL 线程等场景启用，且受 first-party beta / feature gate 控制。
- `hao(messages, keepRecent)` 会收集特定工具的 `tool_use.id`，默认 `keepRecent = 5`。
- 最近 5 个相关工具结果保留，旧的相关工具结果成为清理候选。
- 如果预计节省不足约 `gao = 20000` tokens，则不清理。
- 清理时仍不删除 `assistant tool_use` 或 `user tool_result` block，而是替换旧 `tool_result.content`：
  - 能落盘的结果替换成“Tool result saved to: ... Use Read to view...”。
  - 不能落盘或不适合落盘的结果替换成 `[Old tool result content cleared]`。
- 因为 block 和 `tool_use_id` 还在，协议配对关系不会被破坏。

这和本文前面的结论一致：Claude Code 更偏向“保留会话结构，替换旧工具结果内容”，而不是把旧工具结果整段删除或每轮原样重放。

### 4. 没有看到主循环直接使用 `clear_tool_uses_20250919`

Claude Code binary 内确实包含 Anthropic API 文档文本，里面提到：

```text
context_management.edits: [{ type: "clear_tool_uses_20250919" }]
```

但在 `2.1.195` 可见的压缩代码里，没有看到主循环请求对象直接构造 `context_management: { edits: [{ type: "clear_tool_uses_20250919" }] }`。能确认的是：

- binary 内置文档知道 server-side context editing。
- Claude Code 自身有本地 `<persisted-output>` 替换逻辑。
- Claude Code 自身有 `context_hint` 触发后的旧工具结果清理逻辑。
- Claude Code 另有 autocompact 路径，会在上下文压力较大时总结历史。

因此，不应把“Anthropic API 支持 tool result clearing”直接等同为“Claude Code 每轮都用 `clear_tool_uses_20250919`”。对 Claude Code 2.1.195，更准确的说法是：**它在本地维护完整消息结构，同时通过持久化引用、旧结果占位和自动压缩控制跨轮次上下文体积。**

## 可提炼的设计准则

从 Claude Code 对工具调用上下文的处理方式，可以提炼出一组更通用的 Agent 上下文设计准则。

### 1. 保留结构，不保留全部大型载荷

`assistant tool_use -> user/tool_result` 的协议结构要尽量保留，避免破坏模型对历史工具调用链的理解。但 `tool_result.content` 不必永远保留完整原文。

更合理的做法是：

```text
保留：role、tool_use_id、tool_result block、执行状态、必要预览
替换：大型 raw output、旧过程数据、低价值中间结果
```

也就是说，跨轮次上下文里最重要的是“发生过什么、结果是否可恢复、当前是否还相关”，而不是把所有工具输出原样堆进去。

### 2. 区分可审计历史和模型活跃上下文

工具结果应该进入 trace、文件、对象存储、业务表或专门的 tool result store，用于审计、调试、回放和按需恢复。

但模型下一轮实际看到的上下文应该更克制，只携带当前决策需要的信息：

- 最终结论。
- 必要摘要。
- 业务状态。
- 结果引用。
- 最近且仍相关的少量工具结果。

这条原则的关键是：**持久化完整历史，不等于每轮都把完整历史喂给模型。**

### 3. 大结果默认引用化

大工具输出不应直接长期留在 prompt 中。更好的形态是：

```text
工具结果已保存到 resultRef / 文件路径
这里是前 N 字符预览
需要完整内容时使用 read_tool_result / Read 工具读取
```

这样可以同时满足三个目标：

- 模型有足够预览完成大多数后续推理。
- 用户和系统仍能找回完整原文。
- 上下文窗口不会被单次工具输出撑爆。

### 4. 按最近性和相关性保留

最近的工具结果通常最可能影响下一步行动，因此应优先保留。较旧的工具结果可以转成摘要、占位符或引用。

保留策略不应只按全局时间粗暴截断，还应考虑：

- 工具类型。
- 是否和当前任务链相关。
- 是否已经被 assistant final 消化。
- 是否已经转化成业务事实。
- 是否仍可能被用户追问原文。

例如，最近一次代码评测失败用例、最近一次题目搜索候选、最近一次文件读取结果通常比十轮之前的搜索列表更有价值。

### 5. 替换内容，而不是随意删除消息

清理旧工具结果时，优先替换 `tool_result.content`，而不是直接删除整条消息。

推荐替换形态：

```text
[Old tool result content cleared]

<persisted-output>
Full output saved to: resultRef
Preview:
...
</persisted-output>
```

这样可以避免出现孤立的 `tool_use` 或孤立的 `tool_result`，也能让模型知道“这里曾经有结果，只是内容被清理或引用化了”。

### 6. 分层预算控制

上下文治理不应只有一个全局 token 上限。更稳妥的做法是分层控制：

- 单个工具结果过大：执行后立即持久化并返回预览。
- 单个回合工具结果组过大：请求前替换该组中最大的旧结果。
- 整体上下文接近窗口：触发局部清理或全局 compact。
- 清理收益不足：不要为了形式上的压缩破坏上下文稳定性。

这样可以把问题消化在局部，避免总是在上下文快爆掉时才做一次大规模总结。

### 7. 替换结果要稳定、可复用

同一个 `tool_use_id` 的结果一旦被替换，后续轮次应复用同一个替换内容或同一个 resultRef。

不要让同一段历史在不同轮次之间反复出现以下变化：

```text
raw output -> 摘要 A -> raw output -> 摘要 B -> 占位符
```

稳定替换有两个好处：

- 模型看到的历史更一致。
- prompt caching 的前缀更稳定。

### 8. 清理必须显式可见

不要静默删除工具结果。清理后的占位符应该明确表达：

- 原始内容已被清理。
- 是否仍可恢复。
- 从哪里恢复。
- 使用哪个工具恢复。

这能避免模型误判为“工具没有返回结果”或“结果为空”，也方便用户追问“刚才那个原始输出是什么”。

### 9. 可恢复性优先于摘要漂亮

摘要适合帮助模型快速理解结果，但摘要不等于原始事实。对于可能被追问、审计或调试的结果，最好保留 resultRef。

更好的组合是：

```text
摘要 / 预览：服务当前推理
resultRef：服务后续恢复
raw result store：服务审计和回放
```

不要只保留一个看似流畅但不可验证的摘要。

### 10. 不同工具采用不同上下文策略

工具结果的处理策略应按类型区分，而不是全局一刀切：

- `Read` / `Grep` / `Bash`：大输出适合落盘预览和按需读取。
- 搜索工具：长期保留候选摘要，少保留完整搜索列表。
- 评测工具：保留状态、失败用例摘要、提交 ID，原始日志引用化。
- AI Review 工具：正式结论落业务表，raw result 放 trace。
- 图片、文档、多模态结果：需要单独的媒体预算和降级策略。
- 敏感工具结果：只保留脱敏状态、业务 ID 或安全摘要。

工具类型不同，跨轮次价值、敏感性、体积和可恢复方式都不同。

### 11. 自动压缩是兜底，不是唯一机制

全局 compaction 很重要，但它不应该是唯一的上下文治理手段。

更好的顺序是：

```text
工具结果生成时先控大小
请求组装前做局部替换
上下文压力较大时清理旧结果
最后再做全局 compact
```

这样能减少大规模 compact 对历史细节和缓存稳定性的冲击。

### 12. 可观测性必须内建

工具结果被持久化、替换、清理或压缩时，应记录可观测事件。

至少应记录：

- 哪些 `tool_use_id` 被处理。
- 原始大小和替换后大小。
- 预计节省 token。
- 触发原因。
- 是否保留 resultRef。
- 是否可重新读取。
- 是否影响当前回合或历史回合。

上下文治理如果不可观测，线上排查时很难判断模型到底看到了什么。

### 总结

Claude Code 的优秀设计可以概括为：

```text
保留可验证的对话骨架，
剥离不必要的大型工具载荷；
原始结果持久化，
活跃上下文引用化；
最近结果优先保留，
旧结果可恢复清理。
```

这套原则适合迁移到各类 ReAct / tool-calling Agent：让系统完整保存事实，让模型只看当前需要的事实。

## 对 Prompt Caching 的影响

### 保留完整 chronological history

如果每一轮都把完整历史按真实顺序追加：

```text
developer/system
user1
assistant tool_call
tool result
assistant final
user2
```

同一个会话内，下一轮请求通常是上一轮请求的长前缀再追加新用户消息。这种情况下，prompt caching 可能较容易命中已有前缀。

但代价也明显：

- 输入越来越长，即使 cached tokens 有折扣，也不是免费。
- 上下文窗口被历史工具结果占用。
- 模型注意力可能被旧过程数据干扰。
- 旧工具结果可能过期或敏感。

### 压缩或清理旧工具结果

如果把旧工具结果替换为摘要、占位符或 resultRef：

```text
developer/system
user1
assistant final
tool result summary / placeholder / resultRef
user2
```

优点是输入更短、更干净，也更安全。缺点是从替换发生的位置开始，旧缓存前缀会变化，因此 prompt caching 会部分失效。

这不是一定坏事。上下文更短通常能降低总成本和延迟，也能让模型聚焦。缓存收益和上下文质量之间需要权衡。

### 动态工具结果放前面最差

错误示例：

```text
上一轮工具结果
developer/system
稳定工具说明
当前用户消息
```

如果动态工具结果放在最前面，每轮一变，后面的稳定系统规则也无法作为相同前缀复用，缓存收益会大幅下降。

正确方向是：

```text
稳定 developer/system
稳定 tool schema / examples
服务端上下文
摘要 / 历史 / 工具结果摘要
当前用户消息
```

如果是原始 tool message，则必须放在真实 transcript 中；如果是为了缓存和压缩重排，则应该以摘要或上下文 section 的形式存在。

## 推荐的三层设计

对 `algo-mentor` 这类学习 Agent，建议采用三层设计。

### 1. Trace / Store 层

保存完整工具结果，包括：

- tool name。
- call id。
- arguments。
- raw result。
- visible result。
- resultRef。
- 执行时间。
- 权限状态。
- 错误信息。

用途：

- 审计。
- 调试。
- 回放。
- 线上问题排查。
- 按需重新读取。

这一层可以保存完整信息，但要遵守安全和隐私脱敏规则。

### 2. Business State 层

把工具结果转化为业务事实。

例如：

- `submit_practice_code_review` 的正式 Review 结果落到 Review 表。
- 学习计划草案落到 draft 表。
- 题库搜索结果可以只保留被选中的候选题。
- LeetCode 反馈可以沉淀为提交状态、错误类型、失败用例摘要。

业务事实比原始 tool result 更适合跨轮次长期使用，因为它更稳定、更小、更可控。

### 3. Active Context 层

下一轮 prompt 只带必要上下文：

```text
稳定规则
场景策略
当前题目/计划
活跃摘要
最近少量历史
必要工具结果摘要或 resultRef
当前用户消息
```

原始工具结果只有在以下情况才进入模型上下文：

- 当前 ReAct 回合还没结束，模型必须读取 tool result。
- 工具结果很短，且和用户下一问直接相关。
- 用户明确追问“刚才工具返回的原始结果里...”。
- 模型通过专门工具按 `resultRef` 请求读取。

## 具体规则建议

可以把跨轮次工具结果策略写成规则：

- 当前 run 内：保留完整 `assistant tool_call -> tool result -> assistant final` 链。
- run 结束后：原始工具结果进入 trace/store。
- 下一轮默认：不带大型 raw tool result。
- 最近一轮短结果：可以按真实历史保留。
- 大结果：替换成摘要、占位符和 resultRef。
- 敏感结果：不进入普通历史，只保留状态和业务 ID。
- 已落库业务结果：下一轮优先引用业务事实，而不是 raw tool result。
- 如果清理：按完整 tool-use group 处理，避免孤立 tool call 或孤立 tool result。
- 如果摘要：明确标注来源、时间和不完整性。

## 对 algo-mentor 的落地建议

当前项目里已有一些基础能力：

- `PromptSlot` 中有 `TOOL_RESULT`，但 Practice Chat 当前主要使用 `HISTORY` 和 `CURRENT_USER_MESSAGE`。
- `AgentLoopRunner` 在 run 内会把 assistant tool calls 和 tool result 追加到本次可变工作上下文。
- `ToolResultCompactor` / `RunMessageCompactor` 已经体现了“工具结果可压缩、不可随意删除破坏配对”的方向。
- `ReadToolResultTool` 支持按需读取大结果。
- AI governance 和 persistence 已经记录 `cachedTokens`、trace 和工具事件。

后续可以进一步明确：

- Practice Chat 下一轮 history 是否包含上一轮 tool_call/tool_result 原文。
- Review tool 成功后，下一轮优先放 Review summary 和 reviewId，而不是 raw tool result。
- 学习计划工具搜索结果只长期保留候选摘要或最终选题。
- 大工具结果默认 resultRef 化，必要时通过 `read_tool_result` 再读。
- Prompt caching 指标按场景观察：完整历史 vs 摘要化历史的成本和延迟差异。

## 面试表达

如果被问“Agent 跨轮次要不要保留工具结果”，可以这样回答：

> 我会区分持久化历史和模型输入上下文。当前 ReAct 回合内，工具调用结果必须按协议跟在对应的 tool call 后面，否则模型无法完成后续推理。但当这一轮结束、用户发起下一条消息时，不应该默认把所有历史工具结果原样塞进 prompt。原始工具结果应保存到 trace 或 tool result store，业务上有价值的结果落到业务表；下一轮模型上下文只带最终回答、必要摘要、业务状态或 resultRef。
>
> 这样做也更利于缓存和成本控制。完整 chronological history 在同一会话内可能提升 prompt caching 命中，因为下一轮是在上一轮前缀后追加；但工具结果会让上下文越来越长，也可能过期或包含敏感信息。压缩或清理旧工具结果会让缓存从变化点后部分失效，但能降低输入长度并提高上下文质量。所以实际工程里要做分层：trace 完整保存，active context 选择性携带。
