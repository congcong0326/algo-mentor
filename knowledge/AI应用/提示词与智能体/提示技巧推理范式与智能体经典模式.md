# Prompt 提示技巧、推理范式与 Agent 经典模式

更新时间：2026-06-29

## 核心结论

Few-shot、Chain-of-Thought、Self-Reflection、ReAct、Plan-and-Solve 这些术语经常被一起讨论，但它们不完全属于同一层。

更清楚的分法是：

- **提示技巧**：关注“单次模型调用怎么写 prompt 更稳定”，例如 Zero-shot、Few-shot、结构化输出约束。
- **推理范式**：关注“模型如何组织思考和解题过程”，例如 Chain-of-Thought、Plan-and-Solve、Self-Consistency、Tree of Thoughts。
- **Agent 行为范式**：关注“模型如何在多轮任务中计划、调用工具、观察结果、修正策略”，例如 ReAct、Reflection / Reflexion、Plan-and-Execute。
- **工程架构能力**：关注“如何把模型接入真实系统”，例如 Tool Calling、RAG、Memory、Workflow、Evals、Observability。

一句话概括：

> Prompt Engineering 更偏单次调用质量；Agent 开发更偏多轮决策、工具调用、状态管理和反馈闭环。提示技巧和推理范式是 Agent 开发的重要基础，但不等于完整的 Agent 系统。

## 术语分层

| 层次 | 典型术语 | 主要解决的问题 |
|---|---|---|
| 提示技巧 | Zero-shot、One-shot、Few-shot、结构化输出、角色设定 | 如何让模型更稳定地理解任务 |
| 推理范式 | CoT、Plan-and-Solve、Least-to-Most、Self-Consistency、Tree of Thoughts | 如何让模型更好地拆解和推理 |
| Agent 行为范式 | ReAct、Reflection / Reflexion、Plan-and-Execute | 如何让模型在多步任务中行动和纠错 |
| 工程架构能力 | Tool Calling、RAG、Memory、Multi-Agent、Guardrails、Evals | 如何把模型接入工具、知识库和业务系统 |

## 提示技巧

### Zero-shot Prompting

Zero-shot 是不给示例，直接要求模型完成任务。

适用场景：

- 任务简单。
- 输出格式不复杂。
- 模型对任务类型已经很熟悉。

示例：

```text
请判断下面这段用户反馈的情绪，只输出：正向、负向或中性。

用户反馈：
这个功能终于能用了，但是加载还是有点慢。
```

这个 prompt 没有示例，完全依赖模型已有能力完成分类。

### One-shot Prompting

One-shot 是给一个示例，让模型模仿输入输出模式。

示例：

```text
请把用户问题改写成更适合搜索题库的关键词。

示例：
输入：我想练一下那种用两个指针从两边往中间扫的题
输出：双指针 对撞指针

现在处理：
输入：我想找一些需要维护窗口最大值的题
输出：
```

一个示例可以帮助模型理解“不要长篇解释，只输出检索关键词”。

### Few-shot Prompting

Few-shot 是给多个示例，让模型从示例中归纳任务规则。这也常被称为 **示例驱动提示**。模型没有更新参数，而是在上下文中临时学习任务模式，这种能力也叫 **In-context Learning**。

适用场景：

- 输出格式比较固定。
- 任务边界容易误判。
- 需要模型模仿特定风格、分类标准或结构化字段。

示例：

```text
你需要判断用户发来的内容属于哪类算法学习请求。
只输出 JSON，不要输出额外解释。

类别只能是：
- hint：用户想要提示
- solution：用户想要完整解法
- review：用户贴了代码，希望检查
- concept：用户询问概念

示例 1：
用户：这题我没思路，能给个提示吗？
输出：{"type":"hint"}

示例 2：
用户：直接给我 Java 代码和复杂度
输出：{"type":"solution"}

示例 3：
用户：这是我的代码，为什么过不了？
输出：{"type":"review"}

现在判断：
用户：滑动窗口和双指针到底有什么区别？
输出：
```

这个例子中，Few-shot 的作用不是教模型算法，而是稳定分类边界和输出格式。

### 结构化输出约束

结构化输出不是一个单独的推理范式，但它是提示词工程里非常常见的技巧。它要求模型按固定结构输出，方便程序解析。

示例：

```text
请把下面的学习目标整理成 JSON。
要求：
- difficulty 只能是 beginner、intermediate、advanced
- weeklyHours 必须是数字
- topics 是字符串数组

用户输入：
我准备 6 周刷完二叉树和动态规划，每周大概 8 小时，现在基础一般。

输出 JSON：
```

真实工程里，最好用 JSON Schema、类型校验或服务端解析兜底，而不是只靠自然语言提示。

## 推理范式

### Chain-of-Thought Prompting

Chain-of-Thought，简称 **CoT**，中文常叫“思维链提示”或“分步推理提示”。它的核心是引导模型不要直接给答案，而是先拆解问题。

典型写法：

```text
请先分步骤分析这个问题，再给出最终答案。

问题：
为什么二分查找中 left <= right 和 left < right 对应的边界处理不同？
```

在面向用户的产品中，更推荐让模型输出“简要推理步骤”或“关键依据”，而不是要求暴露完整内部推理过程。

更适合工程使用的写法：

```text
请先列出 3 个关键判断点，再给出结论。
不要输出冗长推理，只保留用户需要理解的步骤。

问题：
这段二分代码为什么会死循环？
```

CoT 更像推理组织方式，不是完整 Agent。它可以提升单次复杂问题回答质量，但本身不包含工具调用、状态管理或反馈闭环。

### Zero-shot CoT

Zero-shot CoT 是不给示例，只用一句话触发分步分析，例如“请一步步分析”。

示例：

```text
请一步步分析，并给出最终结论。

题目：
给定一个有序数组，如何找到第一个大于等于 target 的位置？
```

它适合快速提升复杂题目的可解释性，但输出可能变长，需要控制格式。

### Plan-and-Solve

Plan-and-Solve 的核心是：**先制定计划，再解决问题**。

它和普通 CoT 的区别是，Plan-and-Solve 更强调先给出整体步骤，再按步骤执行。

示例：

```text
请按下面流程回答：
1. 先给出解题计划。
2. 再按计划逐步解决。
3. 最后总结时间复杂度和空间复杂度。

问题：
如何判断一个链表是否有环，并找到环的入口？
```

Plan-and-Solve 在 Agent 里会进一步发展成 **Plan-and-Execute**：先规划任务列表，再逐步调用工具或执行动作。

### Least-to-Most Prompting

Least-to-Most 的核心是：**先解决简单子问题，再逐步推进到完整问题**。

它适合复杂任务、组合问题和教学场景。

示例：

```text
请用从简单到复杂的方式讲解动态规划：
1. 先说明状态表示。
2. 再说明状态转移。
3. 再说明初始化。
4. 最后把这些部分合成完整解法。

题目：
爬楼梯，每次可以爬 1 或 2 阶，问到第 n 阶有多少种方法。
```

这个方式很适合算法学习，因为它符合“先搭局部概念，再合成完整解法”的认知过程。

### Self-Consistency

Self-Consistency 的核心是：**让模型生成多条推理路径，再从多个候选答案中选择最一致或最可靠的结果**。

示例：

```text
请用 3 种不同思路判断下面代码的时间复杂度。
然后比较三个结论是否一致，给出最终答案。

代码：
for i in range(n):
    j = 1
    while j < n:
        j *= 2
```

它适合数学、逻辑、复杂分类等容易出错的任务。代价是调用成本和延迟更高。

### Tree of Thoughts

Tree of Thoughts，简称 **ToT**，可以理解为把推理过程从一条链扩展成多条分支。模型会探索多个候选思路，再评估哪个更好。

示例：

```text
请为这道题提出 3 种可能解法：
1. 暴力思路
2. 使用哈希表的思路
3. 使用双指针或排序的思路

然后比较它们的正确性、复杂度和实现难度，选择最适合面试的方案。

题目：
两数之和。
```

ToT 不一定要真的实现复杂搜索树。工程实践中，经常只是让模型列多个方案并评估优劣。

## 自我反思与自检

### Self-Critique / Self-Evaluation

Self-Critique 或 Self-Evaluation 是让模型对自己的答案进行检查、评分或找漏洞。

示例：

```text
请先给出答案，然后从以下角度自检：
1. 是否回答了用户问题。
2. 是否有不准确或过度推断。
3. 是否遗漏边界情况。

如果发现问题，请给出修正版。

问题：
HashMap 在 Java 里为什么查询通常是 O(1)？
```

这种模式适合写作、代码解释、知识总结和结构化输出检查。

### Generate-then-Verify

Generate-then-Verify 是“先生成，再验证”。它可以由同一个模型完成，也可以由另一个模型、规则引擎或测试用例完成。

示例：

```text
第一步：生成一个 Java 解法。
第二步：用下面的测试用例手动检查输出是否正确。
第三步：如果发现错误，修正代码。

题目：
反转链表。

测试用例：
输入：1 -> 2 -> 3
期望输出：3 -> 2 -> 1
```

真实工程里，更可靠的 Verify 应该尽量交给确定性机制，例如单元测试、JSON Schema、权限校验、数据库约束和业务规则。

### Reflection / Reflexion

Reflection 泛指反思。Reflexion 更常用于 Agent 语境，强调把失败经验或自我反馈用于下一轮尝试。

示例：

```text
你刚才给出的解法在用例 [1,1,1], target = 2 上失败了。
请先总结失败原因，再给出修正策略，最后输出修正后的算法。
```

在 Agent 中，Reflection 常见于：

- 工具调用失败后分析原因。
- 答案被测试或评审拒绝后修改策略。
- 多轮任务中把失败经验写入短期记忆。

## Agent 经典行为范式

### ReAct

ReAct 是 **Reasoning + Acting**，也就是“推理 + 行动”。它是工具型 Agent 很经典的范式。

核心循环：

```text
观察任务 -> 思考下一步 -> 调用工具 -> 观察工具结果 -> 再决定下一步
```

示例：

```text
你是一个算法学习 Agent。
当用户要求推荐题目时，你需要：
1. 分析用户目标。
2. 调用题库搜索工具。
3. 根据工具返回结果筛选题目。
4. 给出推荐理由。

用户：
我想练习二分查找，最好是中等难度，给我 5 道题。
```

如果写成抽象执行轨迹，大概是：

```text
用户目标：练习二分查找，中等难度，5 道题
动作：调用 search_problems(tags=["binary-search"], difficulty="medium")
观察：工具返回候选题列表
动作：筛选覆盖 lower_bound、答案二分、旋转数组等类型的题
最终回答：推荐题目 + 推荐理由 + 练习顺序
```

ReAct 的重点不是“输出很长推理”，而是模型能基于环境反馈决定下一步动作。

### Plan-and-Execute

Plan-and-Execute 是 Agent 里很常见的模式：先规划，再逐步执行。

示例：

```text
你是一个学习计划 Agent。
请先生成任务计划，再逐步执行。

用户目标：
4 周内复习数组、链表、二分、动态规划，每周 6 小时。

计划阶段：
1. 分析用户基础和时间。
2. 拆分学习阶段。
3. 搜索每个主题的候选题。
4. 生成每周计划。
5. 输出复习建议。
```

和 Plan-and-Solve 相比，Plan-and-Execute 更偏 Agent，因为它通常会在执行阶段调用工具、读取外部数据或更新状态。

### Reflection Agent

Reflection Agent 会在完成任务后检查结果，如果不满足目标，就调整策略再执行。

示例流程：

```text
任务：生成一份 4 周学习计划
第一次输出：题量过多，不符合每周 6 小时
自检：计划不可执行，单周负载过高
修正：减少题量，增加复盘时间
最终输出：更可执行的计划
```

Prompt 示例：

```text
生成学习计划后，请根据以下标准自检：
1. 每周任务量是否符合用户可用时间。
2. 题目难度是否循序渐进。
3. 是否包含复盘和错题整理。

如果不符合，请先修正，再输出最终版本。
```

Reflection 的价值在于提高稳定性，但它不能替代外部验证。比如“题目是否存在”“用户是否有权限”“计划是否写入数据库成功”仍然需要工程系统保证。

## 工程实现视角

如果说 Agent 实现有几种经典范式，常见会提到：

- **ReAct**：边推理、边行动、边观察。
- **Plan-and-Execute**：先规划任务步骤，再按步骤执行。
- **Reflection / Reflexion**：执行后自检、纠错、重试，并把经验用于下一轮。

工程上判断一个系统属于哪种模式，重点不是 prompt 里有没有写“请先计划”或“请反思”，而是看**控制流由谁驱动**：

- 如果每一轮都由模型决定下一个 tool call，本质更接近 ReAct。
- 如果计划被结构化生成、解析、保存，并由代码按 step 推进，本质更接近 Plan-and-Execute。
- 如果系统有独立的检查、评审、测试、重试或经验沉淀环节，本质更接近 Reflection / Generate-then-Verify。

现实产品通常不是纯范式，而是混合模式。

### ReAct 的典型代码结构

ReAct 通常实现成一个 agent loop。

```java
AgentState state = AgentState.start(userGoal, context);

while (!state.isDone() && state.withinBudget()) {
    AgentDecision decision = model.decide(
            state.messages(),
            toolRegistry.availableTools(),
            state
    );

    if (decision.isFinalAnswer()) {
        return decision.finalAnswer();
    }

    ToolCall toolCall = decision.toolCall();
    ToolResult observation = toolExecutor.execute(toolCall);

    state.recordToolCall(toolCall);
    state.recordObservation(observation);
}

return fallbackAnswer(state);
```

关键组件通常包括：

- `AgentState`：保存用户目标、历史消息、工具结果、预算、失败信息。
- `ToolRegistry`：声明可用工具及参数 schema。
- `ToolExecutor`：执行工具调用，并处理权限、超时、异常和审计。
- `BudgetGuard`：控制最大轮数、最大 token、最大耗时。
- `Trace`：记录模型决策、工具调用、工具结果和最终输出。

ReAct 的优点是灵活，适合搜索、排查、调试、浏览、代码修改等开放任务。缺点是全局计划可能比较弱，如果不加约束，容易走一步看一步。

### Plan-and-Execute 的典型代码结构

Plan-and-Execute 也有循环，但循环粒度不是“模型每轮自由决定下一个 action”，而是“执行计划中的 step”。

```java
Plan plan = planner.createPlan(userGoal, context);
ExecutionState state = ExecutionState.start(userGoal, context, plan);

while (!plan.isComplete() && state.withinBudget()) {
    PlanStep step = plan.nextRunnableStep();

    StepResult result = stepExecutor.execute(step, state);
    state.recordStepResult(step.id(), result);
    plan.mark(step.id(), result.status());

    if (result.requiresReplan()) {
        plan = replanner.replan(userGoal, plan, state);
        state.recordPlanRevision(plan);
    }
}

return synthesizer.finalAnswer(userGoal, plan, state);
```

典型组件会拆得更清楚：

| 组件 | 职责 |
|---|---|
| `Planner` | 把用户目标拆成结构化计划 |
| `Plan` / `PlanStep` | 保存步骤、依赖、状态、预期产物 |
| `StepExecutor` | 执行单个步骤，可能调用工具或运行内层 ReAct |
| `Replanner` | 步骤失败、信息不足或目标变化时重新规划 |
| `Synthesizer` | 根据所有步骤结果汇总最终答案 |

一个计划通常应该是结构化对象，而不是只存在于自然语言聊天记录里：

```json
{
  "goal": "生成 4 周二分查找训练计划",
  "steps": [
    {
      "id": "analyze_goal",
      "title": "分析用户时间和目标",
      "status": "pending"
    },
    {
      "id": "search_basic_problems",
      "title": "搜索二分基础题",
      "status": "pending"
    },
    {
      "id": "search_answer_binary_problems",
      "title": "搜索二分答案题",
      "status": "pending"
    },
    {
      "id": "compose_plan",
      "title": "组织 4 周训练计划",
      "status": "pending"
    }
  ]
}
```

如果代码仍然是普通 ReAct loop，只是在 prompt 里写了“请先制定计划再执行”，更准确地说是 **Plan-guided ReAct**，不是严格的工程版 Plan-and-Execute。真正的 Plan-and-Execute 至少应该把 plan 作为一等对象管理：可解析、可保存、可更新、可观察、可重试。

工程上常见的混合方式是：

```text
外层：Plan-and-Execute 负责拆任务和管理步骤状态
内层：每个复杂 step 用 ReAct loop 调工具完成
步骤后：用 Reflection / Verify 检查结果是否合格
```

例如“生成学习计划”：

```text
Planner:
1. 分析用户目标
2. 搜索候选题
3. 筛选题目
4. 生成计划
5. 自检计划可执行性

执行第 2 步时：
ReAct 判断该搜哪些 tag -> 调 search_problems -> 观察候选不足 -> 换关键词再搜

执行第 5 步时：
Reflection 检查每周题量、难度梯度、复盘安排是否合理
```

### Reflection / Verify 的典型代码结构

Reflection 通常不是单独替代 ReAct 或 Plan-and-Execute，而是包在生成、执行或步骤结果之后。

```java
Feedback feedback = Feedback.empty();

for (int attempt = 1; attempt <= maxAttempts; attempt++) {
    Draft draft = worker.run(task, context, feedback);
    ReviewResult review = evaluator.evaluate(draft, criteria);

    if (review.accepted()) {
        return draft;
    }

    feedback = review.toFeedback();
    trace.recordRevision(attempt, review);
}

return bestEffortAnswer(trace);
```

`Evaluator` 可以是模型，也可以是确定性机制：

- 模型评审：检查答案是否遗漏、是否有逻辑漏洞、是否符合风格。
- 单元测试：验证代码是否通过测试。
- JSON Schema：验证结构化输出能否解析。
- 业务规则：验证题目是否来自题库、用户是否有权限、数据是否可写入。
- 人工审批：在高风险动作前让用户确认。

工程上更可靠的做法是：能用确定性规则验证的，不只依赖模型自检。

### 三种模式的关系

| 模式 | 代码控制流 | 适合任务 | 主要风险 |
|---|---|---|---|
| ReAct | `while` 循环中由模型决定下一步工具调用 | 调试、搜索、代码修改、资料查询、开放式任务 | 容易缺少全局计划，可能反复试探 |
| Plan-and-Execute | 先生成结构化 `Plan`，代码按 `Step` 执行 | 长任务、跨模块改造、学习计划、复杂工作流 | 计划可能过早固定，需要 replan |
| Reflection / Verify | 生成或执行后评审、测试、重试 | 代码修复、结构化输出、内容生成、质量控制 | 如果只靠模型自评，可能自信但错误 |

一个成熟 Agent 往往会组合三者：

```text
Plan-and-Execute 管整体任务
ReAct 管单步工具探索
Reflection / Verify 管质量闭环
```

## 典型产品归类

Claude Code 和 Codex 这类编码 Agent，不适合简单说成“只属于 ReAct”或“只属于 Plan-and-Execute”。从官方文档能确认的是，它们都具备读取代码、编辑文件、运行命令、调用工具、管理上下文、支持计划/技能/子任务等能力。工程上更合理的归类是：**产品化的混合型 Coding Agent**。

| 产品 / 系统 | 更接近的工程模式 | 判断依据 |
|---|---|---|
| Claude Code | ReAct-like 工具循环 + Plan-and-Execute + Reflection / Verify + Subagents | 官方称其为 agentic coding tool，可读代码、编辑文件、运行命令；Agent SDK 暴露同类 agent loop、工具和上下文管理；还支持 subagents、skills、hooks 等工程机制 |
| Codex | ReAct-like 工具循环 + Plan mode / Goal mode + Reflection / Verify + Subagents | Codex 文档说明一次 prompt 后会循环调用模型并执行文件读取、编辑和工具调用；同时支持 `/plan`、goal、验证、skills、MCP、subagents 等工作流 |

所以更准确的说法是：

> Claude Code 和 Codex 的底层交互很像 ReAct / tool-calling loop；当启用计划、目标、任务清单或按步骤执行时，体现 Plan-and-Execute；当它们运行测试、做 code review、根据失败结果修复时，体现 Reflection / Generate-then-Verify；当它们把任务分给多个独立上下文时，体现 Multi-Agent。

### 每种模式对应的产品多吗

从产品形态看：

- **ReAct / Tool-calling loop 最多**。几乎所有能调用工具的交互式 Agent 都会有类似循环，因为这是模型使用外部工具的基础控制流。
- **Plan-and-Execute 也很多，但常作为功能或内部机制出现**。例如“先生成计划”“任务清单”“Plan mode”“按步骤执行”等，不一定直接用 Plan-and-Execute 这个名字宣传。
- **Reflection / Verify 在严肃工程产品里很常见，但纯 Reflection 产品较少**。编码 Agent 的测试、代码审查、失败重试、自动修复，本质上都带有 generate-then-verify 或 reflection 色彩。
- **Multi-Agent 相对更少，也更贵**。它适合并行探索、代码审查、日志分析、资料总结等可拆分任务，但会增加 token 成本、状态合并和冲突处理复杂度。

因此，面试里不要把产品和论文范式一一硬绑定。更好的表达是：

> 现实 Agent 产品通常是混合架构。ReAct 是基础执行循环，Plan-and-Execute 提供更强的任务组织，Reflection / Verify 提供质量闭环，Multi-Agent 用于并行化和隔离上下文。判断一个系统属于哪种模式，要看代码层面的控制流、状态建模、工具调用和验证机制，而不是只看 prompt 文案。

## 容易混淆的点

### Few-shot 不是 Agent 范式

Few-shot 只是提示技巧。它通过示例稳定模型输出，不代表模型具备工具调用、状态管理或自主执行能力。

如果面试被问到，可以这样回答：

> Few-shot 更偏 Prompt Engineering，用于通过示例约束模型行为；Agent 范式更强调模型在多轮任务中基于目标、工具和环境反馈持续行动。

### CoT 也不等于 Agent

CoT 可以让模型分步骤分析，但单独的 CoT 没有“行动”。没有工具调用、环境观察和状态更新时，它仍然只是一次模型调用里的推理增强。

更准确的说法是：

> CoT 是很多 Agent 内部推理的基础能力，但 Agent 还需要 Action、Observation、Memory、Planning 和 Error Handling。

### Reflection 可以是提示技巧，也可以是 Agent 模式

如果只是让模型“回答后自检”，它是提示技巧。

如果系统把失败原因记录下来，并在下一轮任务中改变策略，它就更接近 Agent 的 Reflection / Reflexion 模式。

### Plan-and-Solve 和 Plan-and-Execute 不完全一样

Plan-and-Solve 通常发生在单次回答中：

```text
先列计划 -> 按计划解题 -> 给出答案
```

Plan-and-Execute 更偏 Agent：

```text
先列计划 -> 调用工具执行每一步 -> 根据结果调整计划 -> 汇总答案
```

### Plan prompt 不等于 Plan-and-Execute

如果只是这样写 prompt：

```text
请先制定计划，然后按计划执行。
```

但代码仍然是：

```text
while not done:
  LLM 决定下一步 action
  调工具
  把 observation 放回上下文
```

那么它更准确地说是 **Plan-guided ReAct**。要称为工程上的 Plan-and-Execute，计划最好被建模成结构化对象，并由代码控制 step 执行、状态记录、失败重试和重新规划。

## 面试表达

可以这样概括：

> 我会把 Prompt Engineering 里的技巧分成几层看。Few-shot、Zero-shot 属于提示技巧，用来提升单次模型调用的稳定性；CoT、Plan-and-Solve、Self-Consistency 属于推理增强范式，用来改善复杂问题的拆解和判断；ReAct、Reflection、Plan-and-Execute 更接近 Agent 行为范式，因为它们涉及工具调用、环境观察、失败反馈和多轮执行。真正工程化落地时，还需要结合 Tool Calling、RAG、Memory、Evals、Guardrails 和日志监控，不能只靠 prompt 本身保证系统可靠。

如果结合算法学习产品，可以这样说：

> 例如在算法学习助手里，Few-shot 可以用来稳定判断用户是在要提示、要完整解法还是要代码 Review；CoT 或 Plan-and-Solve 可以用于讲解复杂题目；ReAct 可以让 Agent 根据用户目标调用题库工具推荐题目；Reflection 可以在生成学习计划后自检题量、难度和复盘安排是否合理。
