# RAG 阶段 1：流程全貌复习笔记

整理时间：2026-06-30

## 核心理解

RAG 可以先简单理解成一条“先找资料，再带着资料回答”的工程链路。

大模型本身不一定知道我们的私有文档、项目资料、学习笔记或最新知识。RAG 不直接训练模型，而是先把外部知识加工成可检索的索引；当用户提问时，再从索引中找出相关材料，把材料放进 prompt 上下文，让大模型基于这些材料回答。

从工程实践上看，RAG 通常分成两条链路：

- **Indexing 链路**：离线或异步执行，把文档处理成可以被检索的知识索引。
- **Query 链路**：在线执行，根据用户问题检索材料，并让模型生成答案。

这两个阶段要分开理解。Indexing 关注“知识怎么存进去”；Query 关注“问题来了以后怎么找出来并用起来”。

## 整体流程

```text
Indexing 链路：

原始文档
-> 文档解析
-> 文本清洗
-> chunk 切分
-> embedding 向量化
-> 写入向量库或搜索索引

Query 链路：

用户问题
-> 问题处理或 query rewrite
-> 问题 embedding
-> 检索 topK 候选 chunk
-> metadata 过滤 / rerank 重排序
-> 拼接 prompt 上下文
-> LLM 生成答案
-> 返回答案、引用来源和日志
```

一条最小可用 RAG 不一定每一步都很复杂，但核心闭环不能少：**文档入库、问题检索、上下文增强、模型回答**。

## Indexing 链路：把知识变成可检索索引

Indexing 阶段处理的是知识库里的资料，例如 Markdown、PDF、网页、代码文档、产品手册、学习笔记等。

| 环节 | 输入 | 输出 | 作用 |
|---|---|---|---|
| 文档收集 | Markdown、PDF、网页、数据库记录 | 原始文档集合 | 明确哪些资料要进入知识库 |
| 文档解析 | 原始文件 | 可处理文本 | 从 PDF、Markdown、HTML 中提取正文 |
| 文本清洗 | 解析后的文本 | 干净文本 | 去掉目录噪声、页眉页脚、无意义空白、乱码 |
| chunk 切分 | 长文本 | 多个文本片段 | 把长文档切成适合检索和放入上下文的小块 |
| overlap 处理 | 相邻 chunk | 带少量重叠的 chunk | 避免重要信息刚好被切断 |
| embedding | 每个 chunk 的文本 | chunk 向量 | 把文本转成可做语义相似度计算的向量 |
| 写入索引 | chunk 文本、向量、metadata | 向量库或搜索索引记录 | 支持后续按语义、关键词、权限、来源检索 |

这里最重要的不是“把整篇文档塞进数据库”，而是把文档拆成合适的检索单元。

chunk 太小，可能只有半句话或半个概念，检索到了也不足以支撑回答。chunk 太大，可能混入很多无关内容，导致检索不精准，也会占用更多上下文 token。

metadata 也很关键。一个 chunk 通常不只存文本和向量，还要存来源文件、标题、章节、标签、更新时间、用户或权限信息。后续返回引用、做权限过滤、排查问题，都需要这些信息。

## Query 链路：根据问题找材料并生成答案

Query 阶段处理的是用户的一次提问。

| 环节 | 输入 | 输出 | 作用 |
|---|---|---|---|
| 用户提问 | 用户自然语言问题 | query 文本 | 表达本次要解决的问题 |
| 问题处理 | 原始 query | 改写后的 query 或多个 query | 补全上下文、改写模糊问题、提高检索命中率 |
| query embedding | query 文本 | query 向量 | 用于和 chunk 向量做相似度匹配 |
| 检索 topK | query 向量、过滤条件 | 候选 chunk 列表 | 找出最相关的一批材料 |
| metadata 过滤 | 候选 chunk、权限、标签、时间 | 过滤后的 chunk | 排除用户无权访问或范围不匹配的资料 |
| rerank | 候选 chunk、query | 重新排序后的 chunk | 用更精细的模型或规则把最有用的材料排前面 |
| 上下文拼接 | 选中的 chunk、系统指令、用户问题 | 最终 prompt | 告诉模型应基于哪些资料回答 |
| LLM 生成 | prompt | 答案 | 组织语言、总结材料、回答用户 |
| 返回与记录 | 答案、来源、检索结果 | 用户响应和日志 | 方便用户追溯，也方便后续评估和排查 |

可以把 Query 链路理解成两件事：

1. **检索阶段**负责把正确材料找出来。
2. **生成阶段**负责基于正确材料说清楚。

这两个阶段都会影响最终效果，但问题类型不同。检索没找到正确资料，模型再强也容易瞎答；资料找对了，但 prompt 约束弱或模型没有正确理解材料，也可能答错。

## 哪些环节影响“是否检索到正确材料”

影响检索命中的环节主要在 indexing 和 retrieval 两侧。

在 indexing 侧：

- 文档解析失败，会导致真实内容根本没有入库。
- 文本清洗太粗糙，会把目录、页眉、乱码等噪声写入索引。
- chunk 切分不合理，会让一个完整知识点被拆碎，或让一个 chunk 混入太多主题。
- overlap 太小，边界处的信息可能被截断；overlap 太大，会带来重复和噪声。
- metadata 缺失，会导致无法按来源、权限、主题、时间过滤。
- embedding 模型不合适，会影响语义相似度表达。

在 query 侧：

- 用户问题太短或太模糊，可能无法匹配到正确 chunk。
- query rewrite 不当，可能把原问题改偏。
- topK 太小，容易漏掉关键材料。
- topK 太大，会引入无关 chunk，干扰后续生成。
- 只用向量检索时，对错误码、类名、函数名、精确术语可能不如关键词搜索稳定。
- 缺少 rerank 时，最相关材料可能没有排在最前面。

所以 RAG 效果差时，第一步通常不是怪大模型，而是先看检索结果：正确资料有没有被召回，排名是否靠前，chunk 是否完整。

## 哪些环节影响“模型是否基于材料正确回答”

即使检索到了正确材料，生成阶段仍然可能出问题。

常见影响因素包括：

- prompt 没有明确要求“只能基于上下文回答”。
- 上下文太长，关键材料被无关内容稀释。
- 多个 chunk 之间存在冲突，模型没有被要求说明冲突。
- chunk 顺序不合理，重要材料排得太靠后。
- 没有要求资料不足时回答“不确定”或“无法根据材料判断”。
- 没有要求输出引用来源，用户无法判断答案依据。
- 模型对材料做了过度推断，把“资料中没有说”的内容补成了确定结论。

因此，RAG 的答案质量不只由检索决定，还取决于上下文组织和生成约束。一个好的 prompt 通常会告诉模型：根据提供的资料回答；资料不足时不要编造；关键结论要附来源。

## topK 的直觉

topK 是检索阶段返回候选 chunk 的数量。

- topK 太小：上下文更干净，但可能漏掉关键资料。
- topK 太大：召回更充分，但会增加 token 成本，也可能把无关内容带进 prompt。

所以 topK 不是越大越好。它要和 chunk size、问题类型、上下文窗口、rerank 能力一起调。

一个常见思路是：先召回较多候选，例如 topK=10 或 20，再用 rerank 选出最适合放入 prompt 的 3 到 5 个 chunk。

## RAG 日志应该记录什么

为了排查问题，RAG 系统不能只记录最终答案。至少应该记录：

- 用户原始问题。
- 改写后的 query，如果有 query rewrite。
- 检索参数，例如 topK、过滤条件、检索方式。
- 命中的 chunk ID、标题、来源、相似度分数、rerank 分数。
- 最终放入 prompt 的上下文片段。
- 使用的模型、embedding 模型、耗时和错误信息。
- 最终答案和引用来源。

这些日志的价值在于区分问题来源：是资料没入库、检索没召回、排序不对、上下文太乱，还是模型没有按材料回答。

注意日志中不能记录 API key、访问令牌、完整 Authorization 头，也要避免记录用户隐私内容。

## 一个快速判断框架

当 RAG 回答不理想时，可以按这个顺序排查：

1. 知识库里有没有正确资料。
2. 正确资料有没有被解析和切分进索引。
3. 用户问题有没有检索到正确 chunk。
4. 正确 chunk 的排名是否足够靠前。
5. 放进 prompt 的上下文是否完整、干净、不过长。
6. prompt 是否约束模型基于资料回答。
7. 模型是否忽略、误读或过度推断了资料。

这个顺序很重要。很多时候看起来是“大模型答错”，实际是前面没有把正确材料交给它。

## 面试表达版本

如果面试中被问“RAG 的流程是什么”，可以这样回答：

> RAG 通常分为 indexing 和 query 两条链路。Indexing 阶段会把 Markdown、PDF、网页等原始文档解析成文本，做清洗和 chunk 切分，然后对每个 chunk 生成 embedding，并连同原文片段和 metadata 写入向量库或搜索索引。Query 阶段在用户提问后，会对问题做 embedding，检索 topK 相关 chunk，必要时做 metadata 过滤、hybrid search 或 rerank，然后把选中的材料拼进 prompt，让大模型基于这些上下文生成答案，并返回引用来源。RAG 的效果主要受两类因素影响：一类是检索质量，比如 chunking、embedding、topK、metadata、rerank；另一类是生成质量，比如上下文是否干净、prompt 是否要求基于资料回答、资料不足时是否允许模型承认不知道。
