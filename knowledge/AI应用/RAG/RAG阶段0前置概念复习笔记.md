# RAG 阶段 0：前置概念复习笔记

整理时间：2026-06-30

## 核心理解

RAG 是 Retrieval-Augmented Generation，即“检索增强生成”。它的核心过程可以拆成三步：

1. 从外部知识源中检索与用户问题相关的资料。
2. 将检索到的资料拼接到大模型的上下文中。
3. 让大模型基于用户问题和外部上下文生成回答。

RAG 的关键不是让模型永久学会你的知识，而是在每次推理时，把当前问题需要的外部知识动态提供给模型。模型参数本身通常不会因为一次 RAG 请求而改变。

更精确地说：

> RAG 通过外部检索，把与当前问题相关的知识动态提供给大模型，从而增强当前生成结果的可靠性、可追溯性和知识覆盖范围。

## 为什么需要 RAG

大模型虽然在海量公开数据上训练过，但它不天然知道用户的私有数据、企业内部文档、最新业务规则、个人知识库或实时数据库内容。

如果直接问模型这些内容，它可能出现三类问题：

- 不知道相关事实，但仍然编造答案。
- 使用训练时学到的旧知识，无法反映最新状态。
- 无法引用用户自己的资料来源，回答缺少可追溯依据。

RAG 通过外部知识检索缓解这些问题。它把“知识存储”和“语言生成”分开：知识可以放在数据库、文档系统、搜索引擎或向量数据库里，模型负责理解问题、阅读检索结果并组织回答。

## RAG 和微调的区别

RAG 和 fine-tuning 都能改善模型效果，但解决的问题不同。

| 对比点 | RAG | 微调 |
|---|---|---|
| 生效阶段 | 推理时 | 训练或再训练时 |
| 是否改变模型参数 | 通常不改变 | 会改变 |
| 主要作用 | 补充外部知识 | 调整模型行为、风格或任务模式 |
| 数据更新 | 更新知识库或索引即可 | 通常需要重新训练或继续训练 |
| 适合场景 | 企业知识库、文档问答、产品手册、个人资料库、实时数据查询 | 固定输出格式、领域表达习惯、分类任务、特定语气 |
| 可追溯性 | 可以返回引用来源 | 不天然具备来源引用 |

简单判断：

- 如果问题是“模型不知道某些资料”，优先考虑 RAG。
- 如果问题是“模型知道资料，但总是不按我想要的格式或风格回答”，可以考虑微调。

## 检索增强增强的是什么

“检索增强”增强的是当前请求的上下文，而不是直接增强模型参数。

RAG 中的知识不是被模型永久记住，而是被模型临时看到。每次用户提问时，系统根据问题检索相关资料，把这些资料作为上下文传给模型。模型离开这些上下文后，并不会自动掌握这些私有知识。

这也是 RAG 适合动态知识库的原因：资料变化时，只需要更新外部索引或数据库，不需要重新训练模型。

## 为什么常用向量数据库

RAG 常用向量数据库，是因为很多问题不能只靠关键词匹配解决，而需要语义相似度检索。

例如用户问：

> Java 里怎么保证多个线程不会同时修改同一个变量？

文档中可能写的是：

> synchronized 可以控制临界区访问，避免并发写入造成数据竞争。

用户没有说 `synchronized`，文档也没有写“多个线程不会同时修改”，但两者语义相关。关键词搜索可能召回效果不好，而向量检索更容易找到语义相近的内容。

向量数据库通常负责：

- 存储文本片段的 embedding 向量。
- 根据用户问题向量查找语义相似的文档片段。
- 结合 metadata 做过滤、排序和权限控制。

不过，RAG 不一定必须使用向量数据库。只要存在“外部检索 + 上下文增强 + 模型生成”，就可以认为属于 RAG 思路。

## RAG 的检索方式不止向量检索

RAG 的检索阶段可以使用多种方式：

- 向量检索：适合语义相似度匹配。
- 关键词检索：适合专有名词、编号、精确词汇、错误码。
- 全文索引：适合大规模文档关键词查询，例如 Elasticsearch。
- SQL 查询：适合结构化业务数据。
- API 查询：适合实时外部系统数据。
- 混合检索：结合关键词检索和向量检索，提高稳定性。

例如用户问“我最近 7 天完成了哪些算法题”，直接查业务数据库可能比向量检索更合适。

## 为什么不能把整篇文档直接塞给模型

把整篇文档直接放进上下文通常不是好方案，原因包括：

- 上下文窗口有限，长文档可能放不下。
- token 成本高，文档越长调用越贵、响应越慢。
- 无关内容太多，会干扰模型判断。
- 相关信息被长上下文稀释，模型可能忽略关键片段。
- 多篇文档同时进入上下文时，更容易超过限制。

RAG 的目标不是把所有资料都交给模型，而是把与当前问题最相关、最有支撑力的资料交给模型。

## 为什么要切 chunk

chunk 是 RAG 检索和上下文拼装中的常见基本单位。长文档通常需要先切成多个片段，再分别向量化和检索。

chunk 太大可能带来这些问题：

- 一个片段里混入太多无关内容。
- 检索命中不够精准。
- 占用过多上下文 token。
- 少量 chunk 就可能塞满上下文。

chunk 太小也会有问题：

- 语义不完整，模型看不懂上下文。
- 一个完整概念被切碎。
- 检索到了片段，但片段本身不足以支撑回答。
- 需要召回更多片段才能拼出完整信息。

好的 chunk 策略要在“检索精准”和“语义完整”之间取平衡。常见做法是按标题、段落、代码块、表格等结构切分，并设置一定 overlap，避免关键语义被切断。

## 向量数据库里通常存什么

向量数据库不只是存向量。一个可用的 RAG 系统通常至少需要存三类信息：

- 向量：用于相似度检索。
- 原文片段：检索命中后作为上下文传给模型。
- metadata：用于过滤、排序、引用和权限控制。

常见 metadata 包括：

- 文档 ID。
- 标题。
- 来源 URL 或文件路径。
- 章节。
- 创建时间和更新时间。
- 作者。
- 标签。
- 用户 ID 或组织 ID。
- 权限级别。
- 文档类型。

metadata 在企业或多用户系统里尤其重要。检索不能只看语义相似度，还必须考虑权限、时效性、文档类型和业务范围，否则可能召回用户不该看到或不该使用的资料。

## RAG 不能完全解决幻觉

RAG 可以降低幻觉，但不能彻底消除幻觉。

原因是错误可能出现在多个环节：

- 检索没有召回正确内容。
- 召回了不相关内容。
- 召回内容本身过期或错误。
- 多个资料之间互相冲突。
- 模型没有严格基于资料回答。
- 模型对资料进行了过度推断。
- prompt 没有要求模型在资料不足时承认不知道。

成熟的 RAG 系统通常会增加约束：

- 要求回答必须基于检索结果。
- 资料不足时明确说明无法判断。
- 返回引用来源。
- 对候选文档 rerank。
- 对答案做事实校验。
- 记录检索结果和生成过程，便于排查问题。

## 一个完整 RAG 链路

完整 RAG 通常可以分成 indexing 链路和 query 链路。

Indexing 链路负责把知识变成可检索索引：

```text
文档加载
-> 文档清洗
-> chunk 切分
-> embedding 向量化
-> 写入向量库、搜索引擎或其他索引
```

Query 链路负责回答用户问题：

```text
用户提问
-> 问题处理或 query rewrite
-> 检索候选内容
-> metadata 过滤
-> rerank 重排序
-> 上下文拼装
-> 大模型生成回答
-> 返回答案和引用来源
```

阶段 0 先理解主线即可：RAG 不是一个单点技术，而是一条由数据处理、检索、上下文组织和生成共同组成的 pipeline。

## 容易混淆的点

### RAG 不是训练模型

RAG 中的资料通常不会进入模型参数，而是在推理时作为上下文提供。模型只是临时阅读这些资料。

### 向量检索不是万能检索

向量检索适合语义相似问题，但对精确编号、错误码、专有名词、短关键词不一定稳定。实际系统常常需要关键词检索、结构化查询或混合检索。

### 检索到了不等于回答正确

模型可能没有正确使用资料，也可能过度总结、错误推断或忽略关键信息。所以 RAG 评估要同时看检索结果和最终回答。

### topK 不是越大越好

topK 太小可能漏掉关键资料；topK 太大可能引入噪声、增加成本，并让模型被无关内容干扰。

## 面试表达版本

如果面试中被问“你怎么理解 RAG”，可以这样回答：

> RAG 是检索增强生成。它不是通过训练让模型记住业务知识，而是在用户提问时，从外部知识库中检索相关内容，再把这些内容作为上下文交给大模型生成答案。这样可以让模型使用私有数据、最新数据或可追溯资料回答问题。一个典型 RAG 系统包括文档清洗、chunk 切分、embedding、索引入库、用户问题检索、rerank、上下文拼装和最终生成。它能降低幻觉，但不能完全消除幻觉，因为检索质量、资料质量和模型是否严格基于上下文回答都会影响最终效果。
