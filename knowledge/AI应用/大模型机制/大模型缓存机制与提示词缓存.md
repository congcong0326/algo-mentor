# 大模型缓存机制与 Prompt Caching

更新时间：2026-06-29

## 核心结论

大模型里的“缓存”至少要区分两层：

- **KV Cache**：模型单次推理或生成过程中，缓存 Transformer attention 层里已经算过的 Key/Value 中间状态。
- **Prompt Caching**：API 服务端跨请求复用相同 prompt 前缀的预填充计算结果，从而降低延迟和输入 token 成本。

两者的底层思想一致：前面已经处理过的上下文，如果后续请求仍然以完全相同的前缀开头，就可以复用部分中间状态。但它们不是同一个概念。KV Cache 是模型推理层面的机制，Prompt Caching 是服务端把这种前缀复用产品化后的能力。

## Transformer 为什么能缓存

现代大语言模型通常基于 Transformer。Transformer 的 self-attention 会把每个 token 的隐藏状态投影成三类向量：

```text
Q = Query
K = Key
V = Value
```

生成下一个 token 时，新 token 的 `Q` 会和历史 token 的 `K` 做匹配，再从历史 token 的 `V` 里聚合信息。

假设模型已经处理了：

```text
A B C D
```

此时模型在每一层、每个 attention head 中都已经得到了 `A B C D` 的 Key/Value。如果继续生成 `E`，没有缓存时可能需要重新处理整段：

```text
A B C D E
```

有 KV Cache 后，可以复用 `A B C D` 的 Key/Value，只计算新 token `E` 的相关状态，然后让 `E` 去 attend 已缓存的历史上下文。

所以 KV Cache 的本质是：**缓存历史 token 在各层 attention 中已经计算好的 Key/Value 中间表示**。

它带来的收益是减少重复计算，代价是占用更多显存或内存。上下文越长、层数越多、batch 越大，KV Cache 占用越明显。

## 为什么缓存天然是“前缀缓存”

大多数生成式语言模型是 decoder-only causal Transformer。它的依赖关系是：

```text
第 1 个 token 不能看后面
第 2 个 token 只能看第 1 个
第 3 个 token 只能看第 1、2 个
第 N 个 token 只能看前面 1..N-1 个
```

因此，如果新请求只是追加内容：

```text
旧请求：
A B C D

新请求：
A B C D E F
```

那么 `A B C D` 的中间状态不受 `E F` 影响，可以复用。

但如果中间发生变化：

```text
旧请求：
A B C D E F

新请求：
A B X D E F
```

从 `X` 开始，后面的上下文都变了。即使后面字面上仍然是 `D E F`，它们的前文已经不同，模型内部表示也不同，所以不能直接复用旧的 `D E F` 中间状态。

这就是前缀缓存的核心逻辑：**前缀不依赖后缀，但后缀依赖前缀**。

## OpenAI Prompt Caching 是什么

OpenAI 的 Prompt Caching 可以理解为：服务端自动识别 API 请求中相同的 prompt 前缀，并复用该前缀的预处理结果。

OpenAI 官方文档说明：

- Prompt Caching 对支持模型自动启用，不需要额外付费。
- 命中缓存时可以降低延迟和输入 token 成本。
- 只有 prompt 开头的完全相同前缀才可能命中。
- 静态内容应该放在 prompt 开头，动态用户内容应该放在后面。
- prompt 至少达到约 `1024` tokens 才有缓存资格；低于该长度时 `cached_tokens` 为 `0`。
- messages、图片、tools 列表、structured output schema 都可能参与缓存，但请求之间必须保持一致。
- 响应中的 `usage.prompt_tokens_details.cached_tokens` 可以用来观察本次命中了多少输入 token。
- `prompt_cache_key` 可以帮助把共享长前缀的请求路由到更可能命中的缓存位置。
- `prompt_cache_retention` 可以在支持模型上配置缓存保留策略。

重要的是，Prompt Caching 不会缓存模型最终回答。即使命中缓存，输出 token 仍然会重新生成。缓存只影响 prompt 前缀处理成本，不改变最终响应语义。

## 系统提示词变化会怎样

系统提示词或 developer message 的变化是否导致缓存失效，取决于变化发生的位置。

如果变化发生在很靠前的位置：

```text
旧：
你是 algo-mentor 的算法刷题教练...

新：
你是 algo mentor 的算法刷题教练...
```

从开头很早的位置开始 token 序列就不同，后面的内容虽然看起来大部分相同，但已经不再是“完全相同的前缀”。这种情况通常接近全部失效。

如果变化发生在中间：

```text
旧：
稳定规则 A
稳定规则 B
旧规则 C
动态上下文 D

新：
稳定规则 A
稳定规则 B
新规则 C
动态上下文 D
```

那么 `稳定规则 A + 稳定规则 B` 这段前缀仍可能命中缓存；从 `规则 C` 的变化点开始，后面一般不能复用。因此是部分失效。

如果只是末尾追加：

```text
旧：
稳定规则 A
稳定规则 B

新：
稳定规则 A
稳定规则 B
新增规则 C
```

前面的稳定部分仍然最容易复用。这也是为什么提示词版本迭代时，尽量把常变内容放后面，稳定基线放最前面。

一句话总结：**缓存不是按“整个 prompt 是否一样”判断，而是按“从开头开始连续相同的前缀有多长”判断。**

## 摘要压缩和历史裁剪会怎样

循环对话中经常会压缩摘要、裁剪历史或重排上下文。这些操作也会影响缓存。

比较好的顺序是：

```text
稳定身份与安全规则
稳定教学策略
稳定工具说明
当前题目与计划上下文
会话摘要
历史消息
当前用户消息
```

如果只压缩了 `会话摘要` 或裁剪了 `历史消息`，那么前面的稳定规则和工具说明仍可能命中，变化点之后失效。这是部分失效。

比较差的顺序是：

```text
会话摘要
当前用户消息
稳定身份与安全规则
稳定工具说明
```

如果摘要每轮变化，它会出现在 prompt 开头，导致后面的稳定规则也无法作为相同前缀复用。这会让缓存收益大幅下降。

所以压缩和裁剪本身不是问题，问题在于动态内容被放到了前缀位置。

## 什么情况下利用不上缓存

常见情况包括：

- prompt 总长度低于约 `1024` tokens。
- 系统提示词、developer message 或 examples 每次都在开头变化。
- 在开头放入 `requestId`、当前时间、用户 ID、traceId、实验随机数等动态字段。
- tools 列表或 structured output schema 每次顺序不同或内容不同。
- 历史消息被放在稳定规则之前。
- few-shot examples 每次随机抽取或随机排序。
- 不同业务场景共用一个过细或过粗的 `prompt_cache_key`，导致路由分散或热点溢出。
- 请求间隔太久，缓存已经过期或被驱逐。

## 怎么利用缓存

实践上可以遵循以下顺序：

```text
1. 稳定平台规则
2. 稳定身份说明
3. 稳定工具调用规则
4. 稳定 few-shot examples
5. 稳定 tool schema / structured output schema
6. 服务端校验过但相对动态的业务上下文
7. 模型生成摘要
8. 最近历史
9. 当前用户消息
```

几个具体建议：

- 把长期不变的规则放最前面，不要在其中夹杂本轮动态字段。
- 稳定 section 的文本要尽量少改，避免无意义的标点、空格、顺序变化。
- tools 和 schema 的顺序保持确定性。
- 变化频繁的摘要、历史和当前消息放后面。
- 需要灰度 prompt 时，用 profile version 管理，不要每次请求动态拼接不同提示词。
- 记录 `cached_tokens`、总 input tokens、延迟和模型名，观察真实命中率。
- 对共享长前缀的请求，考虑稳定设置 `prompt_cache_key`。

## 对 algo-mentor 的启发

当前 `algo-mentor` 的 Practice Chat Prompt Assembly 已经比较符合缓存友好的结构：

```text
STATIC_INSTRUCTION   平台与安全基线
SCENARIO_POLICY      教练风格、回复语言、题目聊天策略、Review tool 边界
RUNTIME_CONTEXT      当前训练上下文
MEMORY_SUMMARY       会话摘要
HISTORY              最近聊天历史
CURRENT_USER_MESSAGE 当前用户消息
```

其中：

- `CACHEABLE_STATIC` 适合用于最稳定的身份、安全和平台规则。
- `CACHEABLE_BY_PROFILE` 适合用于某个 prompt profile 下稳定的教练策略、语言策略和工具边界。
- `NO_CACHE` 更适合题目上下文、摘要、历史和当前用户消息。

这个顺序的好处是，即使当前题目、摘要和历史每轮变化，前面的稳定规则仍可能命中缓存。

后续可以优化的方向：

- OpenAI provider 适配层支持 `prompt_cache_key`。
- OpenAI provider 适配层支持 `prompt_cache_retention`，并根据模型能力配置。
- Prompt Assembly 的 metadata 中继续保留 `promptProfile`、section version、content hash、token estimate 和 truncated sections。
- AI governance 指标继续记录 `cachedTokens`，并按场景、profile、模型观察缓存命中率。
- 学习计划生成和代码 Review 后续迁移到 Prompt Assembly 时，也按“稳定前缀在前、动态上下文在后”的方式组织。

## 一个简化类比

可以把 prompt caching 想象成只会复用“从文档开头开始完全一样的部分”的缓存。

```text
旧文档：
A A A A B B C C

新文档：
A A A A B X C C
```

缓存最多只能复用前面的 `A A A A B`。虽然后面还有 `C C` 一样，但因为中间已经不同，后面的 `C C` 处在不同上下文里，不能直接复用。

这就是为什么“稳定内容放前面、动态内容放后面”是 prompt caching 最重要的工程原则。

## 面试表达

如果面试中被问“大模型缓存是怎么回事”，可以这样回答：

> 大模型缓存可以分两层。第一层是模型推理里的 KV Cache。Transformer 的 self-attention 会为历史 token 计算 Key 和 Value，生成后续 token 时可以复用这些 Key/Value，避免反复计算完整上下文。第二层是 API 服务端的 Prompt Caching，它把相同 prompt 前缀的预填充结果跨请求复用。因为 causal Transformer 中后面的 token 依赖前面的 token，而前面的 token 不依赖后面的 token，所以缓存天然适合复用从开头开始完全相同的前缀。
>
> 如果系统提示词开头变了，缓存基本会大幅失效；如果只在后面追加内容，前面的稳定部分仍然可以复用。摘要压缩和历史裁剪也是一样，只要它们放在后半段，前面的稳定规则仍能命中。工程上应该把稳定指令、工具 schema 和 examples 放前面，把用户输入、摘要、历史和本轮上下文放后面，并通过 `cached_tokens` 指标观察真实缓存命中效果。
